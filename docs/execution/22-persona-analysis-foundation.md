## 1. 背景

ClawPlay 当前已经完成了一个可运行的 `SOUL.md` 目录与分享 MVP：

- Soul 浏览、详情、原文查看、下载、复制安装命令已打通
- 投稿、后台审核、发布链路已打通
- SEO、OG、合集、专题页、分享文案、部署与 smoke check 已补齐

但产品表达仍然偏“目录站视角”：

- 用户能看到 Soul 原文
- 用户能复制和下载
- 用户能通过合集快速挑选

却还不够回答一个更本质的问题：

**这份 Soul 到底是什么人格？和别的 Soul 差在哪里？我为什么要把它换到自己的龙虾上？**

结合 OpenClaw 的真实语义，`SOUL.md` 并不是普通 prompt 文本，而更接近 agent 的人格 / 气质 / 边界定义文件。ClawPlay 的下一版本，应该从“目录站”往“人格分享平台”升级。

## 2. 目标

本子项目标：

1. 把 Soul 从“原文文件”升级成“可理解、可比较的人格资产”。
2. 为每个 Soul 建立统一的人格维度分析，并在前台展示雷达图与摘要。
3. 在后台引入 AI 辅助的人格分析流程，提高新 Soul 的理解速度与展示一致性。
4. 在不引入完整 Agent 镜像复杂度的前提下，先把人格层产品能力做深。
5. 为后续 `Soul Pack` / `Agent Template` 的升级预留结构基础。

## 3. 非目标

本子项不做：

- 完整 Agent 镜像打包与一键本地部署
- 用户自由上传完整 workspace 配置包
- 复杂推荐算法或个性化推荐系统
- 社区互动系统、评论、收藏体系
- 标签开放自由创建
- 对所有页面做一次大规模视觉重构

## 4. 用户价值 / 业务价值

### 4.1 用户价值

- 用户第一次看 Soul 时，不再只能靠“标题 + 摘要 + 原文长度”猜人格。
- 用户可以更直观比较“暴躁”“温润”“谋士”“陪伴”这类气质差异。
- 用户能更快判断某个 Soul 适不适合自己，而不是先替换再碰运气。
- 用户投稿后，站点能更稳定地提炼出人格特征，减少展示信息过于随缘的问题。

### 4.2 业务价值

- ClawPlay 的产品定位会从“目录站”升级成“人格分享平台”，更有辨识度。
- Soul 内容将具备更强的传播素材，不只是原文分享，而是人格画像分享。
- 后续做合集、专题、推荐、Soul Pack 时，有统一的人格结构可复用。
- 管理后台能更快理解新稿件，审核效率和一致性更高。

## 5. 子项定位

这一子项对应 `P7 / N2 + N3` 中的人格分析基础设施部分。

它只负责：

- 人格维度定义
- 雷达图 / 性格维度展示
- AI 辅助分析与后台确认
- 人格画像所需的基础数据结构

它不负责整个 `Soul as Persona` 版本的全部范围。完整版本规划见：`docs/execution/20-soul-as-persona-vnext.md`。

## 6. 交付范围

本子项交付内容：

1. 一套公开展示的人格维度体系（雷达图使用）
2. 一套后台审核的人格分析流程（AI 辅助 + 人工可覆写）
3. Soul 详情页新增人格画像模块
4. 后台审核详情页新增人格分析卡片
5. 对现有 11 个 Soul 的人格分析回填
6. README / 文案 / 路线图同步
7. 一份技术实现设计文档

## 7. 方案设计

### 7.1 与版本总规划的关系

这个子项完成后，ClawPlay 才能具备“人格分析与结构化展示”的底层能力。

但它本身并不自动完成：

- 全站文案重构
- Soul Pack manifest
- 人格导向合集

这些内容仍由 `21-soul-as-persona-vnext.md` 统筹。

### 7.2 公开展示的人格维度

前台使用统一的 6 维人格雷达，不展示“总分”，只展示人格轮廓。

|维度 |含义 |低分表现 |高分表现 |
|---|---|---|---|
|主动性 |是否主动推进、拆解和建议下一步 |等待指令、少推进 |主动拆解、主动建议、主动收口 |
|温度感 |是否带陪伴感、安抚感、亲近感 |冷静克制、低情绪陪伴 |温暖贴近、安抚感强 |
|锋利度 |是否直接、犀利、会指出问题 |委婉圆润 |直给、挑错、批判感强 |
|角色感 |是否有鲜明人格语气和角色氛围 |工具型、中性 |人格明显、角色表达强 |
|边界感 |是否坚持设定、原则、限制 |适配性强、随和 |原则清晰、拒绝与边界明确 |
|结构感 |输出是否有框架、层次、秩序 |发散、灵感型 |条理清晰、结构稳定 |

### 7.3 后台内部维度

后台不只看公开人格，还要看审核效率相关信息。

内部保留 4 个不对外直接展示的辅助维度：

- 完整度：原文是否足够支撑人格判断
- 一致性：原文、摘要、预览三件套是否互相一致
- 可发布性：前台展示是否清晰可理解
- 风险度：是否存在来源、授权、内容边界问题

这 4 个维度主要用于后台提示，不用于前台雷达图。

### 7.4 评分原则

- 公开人格维度使用 `0-100` 内部数值存储。
- 前台默认展示雷达图 + 维度标签文案，不强调“高分更强”。
- 人格分析必须带“简短理由”和“一句话总结”。
- AI 结果默认是**建议值**，管理员可人工调整后确认。

### 7.5 AI 分析原则

AI 的任务不是判断 Soul 好不好，而是回答：

- 这是什么人格
- 在 6 个维度上大致是什么轮廓
- 为什么做出这样的判断

分析输入权重建议：

- `rawSoul`：70%
- `previewHook / previewPrompt / previewResponse`：20%
- `title / summary / tags / tones`：10%

### 7.6 信息结构变化

#### Soul 详情页

新增“人格画像”模块，位置建议放在详情页主要内容区靠前位置，顺序为：

1. 一句话人格定位
2. 6 维雷达图
3. 适合谁 / 不适合谁（第一版可由 AI 生成建议，后台人工确认）
4. 维度理由摘要
5. 原始 `SOUL.md`
6. 下载 / 复制 / 安装说明

#### 后台审核详情页

新增“人格分析”卡片，支持：

- 查看 AI 分析状态
- 重新分析
- 接受 AI 建议
- 人工微调维度
- 标记为已确认

### 7.7 当前版本边界

为了控制范围，本版本只把人格层做深，不继续上更重的“模板层”：

- 先不做 Soul Pack 打包
- 先不做完整 Agent Template
- 先不做跨页面的大规模推荐系统

这样可以确保本版本仍然聚焦在 Soul 本身的理解和表达。

## 8. 风险与取舍

### 风险 1：把雷达图做成“优劣排名”

如果用户把人格维度误解成“高分更强”，会把产品带偏成排行榜。

**取舍：**

- 不展示总分
- 不展示强弱排名
- 只强调人格轮廓与适配关系

### 风险 2：AI 误判人格

AI 可能把风格化文本误判为真实边界，或把预览语气误判为原文主气质。

**取舍：**

- 必须保留人工覆写
- 分析结果保存理由与置信度
- 发布阶段允许人工确认后再作为最终展示值

### 风险 3：版本范围过大

如果同时推进人格维度、详情页大改、列表页改版、Soul Pack、模板化安装，会明显超范围。

**取舍：**

- 本版本只做“人格层基础设施 + 一个主展示页面 + 一个后台页面”
- 列表页只做轻量升级，不做重改

### 风险 4：现有静态 Soul 与发布 Soul 存储不一致

当前站内既有静态文件 Soul，也有数据库发布 Soul。

**取舍：**

- 本版本统一引入独立的人格分析存储层，不直接绑死到某一种 Soul 来源

## 9. 实施拆解

1. 确认人格维度定义与命名
2. 设计分析结果的数据结构
3. 建立 AI 分析与审核确认流程
4. 完成后台人格分析卡片
5. 完成 Soul 详情页人格画像模块
6. 批量回填现有 11 个 Soul
7. 补 README、文案与路线图同步

## 10. 验收标准

满足以下条件即可视为本版本完成：

1. 至少一个 Soul 详情页可以展示完整人格画像模块
2. 前台雷达图成功显示 6 维人格轮廓
3. 后台审核详情页可以查看 AI 人格分析建议
4. 管理员可以接受或覆盖 AI 给出的维度值
5. 新投稿可以进入人格分析流程
6. 现有 11 个 Soul 已完成回填或具备批量回填脚本
7. README 已同步下一版本路线图

## 11. 上线与回滚

### 上线顺序

1. 先落数据库结构与 AI 分析存储
2. 再落后台分析卡片
3. 再落前台详情页人格模块
4. 最后回填存量 Soul 并同步文档

### 回滚策略

- 前台人格模块可单独隐藏，不影响原有 Soul 浏览 / 下载 / 投稿链路
- AI 分析服务异常时，可退回人工空白状态，不阻塞站点主流程
- 新表 / 新字段为增量结构，回滚时不影响旧数据读取
