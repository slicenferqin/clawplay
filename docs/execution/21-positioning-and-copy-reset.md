## 1. 背景

`P7 / Soul as Persona` 已经明确：下一版本的关键，不是继续多做几个页面，而是先把 ClawPlay 的产品语义校正过来。

当前站点虽然已经能跑通：

- Soul 浏览
- Soul 详情
- 原文查看 / 下载 / 复制
- 投稿 / 审核 / 发布
- 合集 / 专题 / 分享

但站点的主叙事仍然明显偏向：

- “安装一个 Soul”
- “下载一个 `SOUL.md`”
- “复制命令装进 OpenClaw”

这套表达在 MVP 阶段没问题，因为它帮我们先把主链路打通了。

但进入下一版本后，如果不先做认知校正，就会出现一个很大的问题：

**产品已经想往“人格分享平台”走，但用户还会继续把我们理解成一个 Prompt 目录站。**

所以 `N1` 的任务，不是润色文案，而是**重写产品定位**。

## 2. 目标

本子项目标：

1. 完成 ClawPlay 的全站认知校正，让站点统一从“安装 / 下载 / 提示词”转向“灵魂 / 人格 / 导入 / 替换”。
2. 让首页、About、详情页、投稿页、安装页使用同一套术语体系与产品叙事。
3. 明确区分“当前仍然是 Soul 层产品”与“未来可能演进成 Soul Pack / Agent Template”的边界。
4. 为后续 Schema v2、详情页重构、Soul Pack 草案提供稳定的表达基线。

## 3. 非目标

本子项不做：

- 新增复杂业务能力
- 重做整站视觉风格
- 引入新的 AI 分析接口
- 修改投稿 / 审核 / 发布的数据结构
- 直接实现 Soul Pack 导入器

本子项只解决：

**用户应该如何正确理解 ClawPlay。**

## 4. 用户价值 / 业务价值

### 4.1 用户价值

- 用户会更快理解 Soul 的价值不只是“文件”，而是“人格”。
- 用户不会再把 ClawPlay 误解成一个普通 prompt 收录站。
- 用户在浏览、选择、导入 Soul 时，心理模型会更稳定。
- 投稿人更容易理解自己在分享什么：不是一段文本，而是一种灵魂人格 preset。

### 4.2 业务价值

- 产品定位更清晰，传播时更容易被记住。
- 后续 Schema、详情页、合集升级会有统一的语义约束，不会每个页面各说各话。
- README、About、首页 Hero、投稿引导之间不会互相打架。
- 这一步完成后，后面做人格分析和详情页重构时，产品叙事不会再反复摇摆。

## 5. 核心问题定义

本子项主要解决 3 个问题：

### 5.1 ClawPlay 到底是什么

应该从：

- 中文 `SOUL.md` 目录与分享站

逐步升级为：

- 面向 OpenClaw 的中文 Soul 人格分享平台
- 一个帮助你发现、比较、替换龙虾灵魂人格的入口

### 5.2 Soul 到底是什么

应该从：

- 一份原文文件
- 一段提示词
- 一个安装对象

逐步升级为：

- 一种人格 preset
- 一层身份 / 气质 / 边界定义
- 一个可比较的灵魂对象

### 5.3 “安装”到底意味着什么

当前产品里，“安装”这个词容易被误解成：

- 完整 Agent 一键部署
- 完整工作流克隆

但事实并不是这样。

所以文案要更准确地引导到：

- 导入起始 Soul
- 替换当前 Soul
- 复制 / 下载原始 `SOUL.md`

而不是制造“完整镜像一键装好”的错觉。

## 6. 统一术语体系

### 6.1 推荐主术语

优先使用：

- 灵魂
- 人格
- 起始 Soul
- 导入
- 替换
- Soul preset
- 人格画像
- 灵魂人格

### 6.2 应减少或谨慎使用的术语

- 提示词
- 一键安装
- 系统提示词
- 模板商店
- 完整镜像
- 一键部署 Agent

### 6.3 术语策略

不是一刀切禁用旧词，而是分场景收口：

- 技术说明页可以保留“原始 `SOUL.md`”
- 操作页可以保留“复制命令”
- 但主叙事层尽量使用“人格 / 灵魂 / 导入 / 替换”

## 7. 页面级改写范围

### 7.1 首页

目标：从“目录站首页”升级为“人格分享平台首页”。

需要调整的重点：

- Hero 标题与副标题
- Journey 三步说明
- 首页模块标题
- 合集入口的语义表达

首页要明确表达：

- 你不是在挑一段文本
- 你是在挑一种灵魂人格
- ClawPlay 帮你先理解它是谁，再决定是否导入

### 7.2 About 页

目标：把“项目说明页”升级为“产品定位说明页”。

About 要明确讲清：

- ClawPlay 为什么存在
- `SOUL.md` 为什么不只是普通 prompt
- 为什么当前只做 Soul 层，而不是完整 Agent 镜像
- ClawPlay 后续会如何从 Soul 走向 Soul Pack / Agent Template

### 7.3 Soul 详情页

目标：在正式做详情页结构重构前，先把语言校正过来。

需要优先调整：

- 下载 / 安装 / 复制相关措辞
- 原始 SOUL 展示区说明
- 侧边说明块和 CTA

### 7.4 投稿页

目标：让投稿人理解自己在分享的是“人格资产”。

需要明确传达：

- 现在只有一份原始 `SOUL.md` 也可以先投
- 你提交的是一个灵魂人格 preset
- 后台会帮助补全展示层信息

### 7.5 安装页

目标：从“安装指南”转成“导入 / 替换说明”。

安装页仍然保留命令和回滚说明，但主叙事要改成：

- 先理解这个 Soul
- 再决定是否导入
- 不喜欢就换回原来的灵魂

## 8. 交付范围

本子项交付内容：

1. 一份统一术语词表
2. 首页 / About / 投稿页 / 安装页 / Soul 详情页文案策略
3. 页面级改写清单
4. 关键标题、副标题、按钮文案的重写建议
5. README 产品定位同步
6. 一份实施设计文档

## 9. 方案设计

### 9.1 先统一产品母叙事，再改页面局部文案

本子项不建议直接“看见哪里别扭就改哪里”，而应该先定母叙事：

> ClawPlay 是一个面向 OpenClaw 的 Soul 人格分享平台。\
> 你在这里不是单纯下载 `SOUL.md`，而是在发现、比较、导入一种灵魂人格。

所有页面的文案都应该围绕这句母叙事延展。

### 9.2 分层表达

建议把表达分成三层：

#### A. 产品定位层

用于首页 Hero、README、About 顶部：

- 灵魂人格
- 人格分享平台
- Soul preset

#### B. 选择决策层

用于详情页、合集页：

- 适合谁
- 边界与气质
- 是否导入到自己的龙虾

#### C. 操作执行层

用于安装页、详情页操作区：

- 复制命令
- 下载原始 `SOUL.md`
- 替换当前 Soul
- 导入起始 Soul

### 9.3 保持“准确”优先于“营销”

文案不能为了更热闹而说大话。

例如：

- 不要说“完整复刻 Agent”
- 不要说“一键部署你的同款龙虾人格系统”
- 不要暗示当前已经具备 Soul Pack / 镜像能力

要坚持：

- 当前阶段先做 Soul 层
- 这是灵魂人格的导入 / 替换
- 后续才会逐步走向更完整模板层

## 10. 风险与取舍

### 风险 1：文案过猛，功能承接不上

如果文案说得太大，而页面结构没跟上，用户会觉得“说得高级，结果还是下载文件”。

**取舍：**
- 先把语义重写到“准确但克制”
- 不越过当前能力边界做承诺

### 风险 2：术语改太猛，老用户看不懂

如果完全禁用“安装”“下载”，反而会让用户不知道怎么操作。

**取舍：**
- 操作层保留必要旧词
- 定位层与解释层优先切换到新词

### 风险 3：每页各自发挥，语气不统一

**取舍：**
- 必须有统一术语表
- 必须有页面级 copy brief
- 实施时以文档为准，不现场即兴

## 11. 实施拆解

1. 建立术语词表与替换原则
2. 为首页、About、投稿页、安装页、详情页分别写 copy brief
3. 统一 metadata 描述策略
4. 更新 README 产品定位
5. 改写页面文案
6. 做一次全站语言回归检查

## 12. 验收标准

满足以下条件即可视为完成：

1. 首页、About、投稿页、安装页、Soul 详情页的主叙事已统一
2. 关键入口的措辞已从“提示词 / 一键安装”收口到更准确的产品语言
3. README 产品定位与站内表达一致
4. 用户读完首页与 About 后，能理解 ClawPlay 现在做的是 Soul 人格层产品
5. 所有改写后的文案不越过当前能力边界
