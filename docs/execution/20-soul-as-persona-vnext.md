## 1. 背景

ClawPlay 这一期已经完成了 `SOUL.md` 目录与分享 MVP：

- Soul 浏览、详情、原文查看、下载、复制命令可用
- 投稿、审核、发布主链路可用
- 合集、专题页、分享文案、SEO、OG、部署链路可用

这说明一件事已经被验证：

**OpenClaw 的 Soul 这一层，确实值得被浏览、比较、投稿和传播。**

但当前产品表达仍然偏“目录站”与“安装链路”：

- 更像在展示一份原文文件
- 更像在告诉用户“怎么复制、怎么下载、怎么替换”
- 还没有充分表达 `SOUL.md` 作为人格 / 气质 / 边界资产的价值

从 OpenClaw 的真实语义出发，下一版本不应该继续优先堆更多页面，而应该把 Soul 从“文件”升级成“人格对象”。

所以，ClawPlay 的下一版本主题应该明确为：

**Soul as Persona**

也就是：

**让用户不只是下载一份** `SOUL.md`，而是真正理解、比较并选择一种灵魂人格。

## 2. 目标

本版本目标：

1. 把 ClawPlay 的产品定位从“目录与安装入口”升级为“人格分享平台”。
2. 把 Soul 从偏目录字段的内容对象，升级为带人格结构的可比较对象。
3. 把 Soul 详情页从“原文 + 下载 + 复制”升级为“结构化人格详情页”。
4. 为后续 `Soul Pack` / `Agent Template` 铺设数据与产品接口，但不提前做重型打包链路。
5. 让合集、推荐与传播逐步从“功能导向”过渡到“人格导向”。

## 3. 非目标

本版本不做：

- 完整 Agent 镜像打包与一键本地部署
- 完整 workspace 模板商店
- 复杂推荐算法或个性化 feed
- 社区互动系统、评论、收藏体系
- 用户自由创建公开标签
- 大规模重做整站视觉风格

## 4. 用户价值 / 业务价值

### 4.1 用户价值

- 用户看 Soul 时，不再只是看标题和原文，而能理解“这是谁”。
- 用户能更直观比较暴躁、温润、粘人、谋士、陪伴、毒舌等人格差异。
- 用户能更快判断一个 Soul 是否适合自己，而不是先换上去再碰运气。
- 用户能逐步理解 Soul 与未来更完整的 Soul Pack / Agent Template 的关系。

### 4.2 业务价值

- ClawPlay 会从“目录站”升级成一个更有辨识度的人格层产品。
- 内容本身会具备更强的结构化资产属性，后续更容易做合集、专题、传播和推荐。
- 为未来的 Soul Pack / Agent Template 演进提供真实的数据与信息结构基础。
- 产品叙事会更贴近 OpenClaw 的本质，而不是停留在“prompt 站”的误解里。

## 5. 版本主题与命名

建议版本名：

**ClawPlay vNext：Soul as Persona**

中文可称：

**人格化升级版**

本版本只做一件事：

**让 Soul 成为可理解、可比较、可传播的人格对象。**

## 6. 核心工作流 / 工作流分解

本版本不是一坨大改，而是 5 条明确的工作流：

### N1：定位与文案重构

目标：完成全站认知校正。

#### 要解决的问题

当前用户仍可能把 ClawPlay 理解成：

- 一个提示词目录
- 一个下载 `SOUL.md` 的地方
- 一个“安装 Soul”的工具页

这会把产品讲浅。

#### 本工作流要完成的事

- 首页、About、详情页、投稿页统一改写
- 从“安装一个 Soul”转成：
  - 发现一种人格
  - 看清它的边界和气质
  - 决定是否导入到自己的龙虾
- 用词统一收口到：
  - 灵魂
  - 人格
  - 起始 Soul
  - 导入
  - 替换
  - Soul preset

#### 验收结果

- 主要页面文案叙事统一
- “提示词 / 一键安装”这类误导性表述明显减少
- 用户能理解 ClawPlay 在做“人格资产”而不是普通 prompt 站

### N2：Schema v2

目标：把 Soul 升级成“人格对象”。

#### 本工作流要完成的事

在现有目录字段基础上，补齐人格结构字段，例如：

- `archetype`：谋士 / 暴躁 / 粘人 / 温润 / 陪伴 / 审查官
- `tone`：冷 / 暖 / 克制 / 锋利 / 俏皮
- `initiative`：被动 / 适中 / 主动
- `warmth`：低 / 中 / 高
- `aggression`：低 / 中 / 高
- `clinginess`：低 / 中 / 高
- `fitFor`
- `notFitFor`
- `coreTruthsSummary`
- `boundariesSummary`
- `vibeSummary`
- `continuitySummary`
- `recommendedModel`
- `recommendedSkillsOrTools`

并建立结构化的人格分析结果存储层。

#### 验收结果

- 新旧 Soul 都能承载人格结构字段
- 投稿、后台、详情页都能读取这套结构
- 不再只依赖 `rawSoul` 作为唯一展示核心

### N3：Soul 详情页重构

目标：把详情页从“目录详情”升级成“人格详情”。

#### 目标顺序

详情页信息结构建议改为：

1. 一句话人格定位
2. 适合谁 / 不适合谁
3. 性格维度条或雷达图
4. `Core Truths / Boundaries / Vibe / Continuity` 摘要
5. 示例对话 / 行为预期
6. 原始 `SOUL.md`
7. 下载 / 复制 / 导入说明

#### 验收结果

- 用户先理解“这是谁”，再去看原文
- 详情页对 Soul 的人格差异表达更强
- 原文不再是唯一主角，而是结构化人格之后的补充证据

### N4：Soul Pack 草案

目标：为后面的模板层铺地基，但不提前上重型安装链路。

#### 本工作流要完成的事

先定义一个最小可行的 `Soul Pack` manifest 概念，包含：

- `SOUL.md`
- metadata
- 推荐模型
- 推荐 skills / tools
- 安装说明
- 示例对话

#### 验收结果

- 产品与数据层都知道 `Soul Pack` 将来长什么样
- 但当前版本不要求真正可一键导入
- 为未来的 `Agent Template` 留出升级接口

### N5：新版合集

目标：让合集从“用途导向”逐步过渡到“人格导向”。

#### 当前基础

现在已有：

- 新手首选
- 开发首选
- 当前热门
- 最近新增

#### 本工作流要完成的事

在保留现有入口的基础上，继续扩成更有人格感的专题，例如：

- 暴躁系
- 温润系
- 谋士系
- 陪伴系
- 毒舌系

#### 验收结果

- 合集不再只是功能导航
- 逐步变成 Soul 人格传播入口
- 后续更适合做社群分享与外部传播

## 7. 本版本的交付范围

本版本最终应交付：

1. 全站定位与文案重构
2. Soul Schema v2
3. Soul 详情页结构化重做
4. 人格分析与雷达图模块
5. `Soul Pack` manifest 草案
6. 人格导向的新版合集规划
7. README、路线图和文档体系同步

## 8. 设计原则

### 8.1 先做语义正确，再做功能扩张

如果产品语义不对，做再多页面也会把用户带到错误理解上。

### 8.2 先做人格层，不急着做模板层

`SOUL.md` 是当前最适合切入、最适合传播、最适合社区化的一层。

### 8.3 先做结构化表达，不急着做复杂推荐

只有当 Soul 被结构化成“人格对象”，后续推荐、专题、比较才有坚实基础。

### 8.4 先做轻量 Soul Pack，不做完整镜像

未来方向可以是 Agent Template，但当前阶段更适合先定义结构，而不是强推重型导入链路。

## 9. 风险与取舍

### 风险 1：范围蔓延

如果 N1~N5 同时全面展开，版本会明显过重。

**取舍：**

- 优先级按 `N1 > N2 > N3 > N4 > N5`
- 先做核心认知与结构，再做衍生扩展

### 风险 2：把 Soul 讲成“完整 Agent”

如果叙事过度，会再次掉回“镜像 / 一键复刻”的误解。

**取舍：**

- 当前版本明确只覆盖人格层
- `Soul Pack` 只做草案，不做完整部署承诺

### 风险 3：结构化字段过多，导致投稿劝退

如果把全部人格字段都丢给用户填，会明显增加投稿成本。

**取舍：**

- 投稿前台先不强推用户填写全部人格字段
- 优先走后台 AI 建议 + 人工确认
- 后续再决定哪些字段适合开放给投稿人补充
