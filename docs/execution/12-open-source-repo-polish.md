## 1. 背景

ClawPlay 现在已经不只是一个本地开发项目，而是开始进入公开被看到的阶段。

仓库已经出现了第一颗 star，这说明外部用户已经开始把 GitHub 仓库当成项目入口来判断：

- 这是不是一个认真在做的项目
- 这个项目现在进行到哪一步
- 我该怎么本地跑起来
- 我应该通过网站投稿，还是通过 PR 贡献
- 这个项目对安全问题、配置说明、开源边界是不是清楚

在域名备案仍未完全放开的窗口期，GitHub 仓库其实就是 ClawPlay 最重要的公开门面之一。

如果仓库门面与产品现状不一致，会直接影响：

- 外部用户的信任感
- star 到 issue / discussion / PR 的转化
- “这个项目是不是还在维护”的判断
- 后续开源协作效率

## 2. 目标

本阶段目标：

1. 让仓库首页描述与当前网站产品形态保持一致。
2. 让贡献路径从“纯 PR 提交内容”升级为“网站投稿 + GitHub PR 双路径”。
3. 补齐基础开源可信度文件，如安全说明和环境变量示例。
4. 降低外部用户第一次了解、运行和参与 ClawPlay 的成本。

## 3. 非目标

本阶段不做：

- 重做整站产品文案
- 新增社区机制或复杂治理规则
- 引入 CI、自动发布或复杂 GitHub Actions 流水线
- 做重型开源规范体系

本阶段只做“高性价比、立即提升专业度”的仓库门面收口。

## 4. 用户价值 / 业务价值

### 用户价值

- 第一次进入仓库时，能快速理解 ClawPlay 是什么。
- 能清楚区分“网站投稿”和“代码/文档贡献”的路径。
- 能快速在本地运行，不需要到处猜环境变量。

### 业务价值

- 提升仓库的专业感和可信度。
- 降低开源入口的认知摩擦。
- 为备案放开后的第二波曝光提前准备好公开门面。

## 5. 交付范围

本阶段建议交付：

1. 重写 `README.md`
2. 更新 `CONTRIBUTING.md`
3. 新增 `.env.example`
4. 新增 `SECURITY.md`
5. 补一份实施记录文档

## 6. 方案设计

### 6.1 README 改为网站优先

README 不再以“人格文件仓库”为中心，而是以“正在运行的网站产品”来组织：

- 项目定位
- 当前能力
- 在线入口
- 本地启动方式
- 环境变量
- 常用命令
- 贡献路径
- 文档索引

### 6.2 贡献路径双轨化

明确区分：

- **内容投稿**：优先走站内投稿页
- **代码 / 文档 / bug 修复**：走 GitHub PR

这样更符合 ClawPlay 当前的真实产品形态，也更符合后续扩展方向。

### 6.3 开源可信度补齐

补齐两个最基础但最有价值的文件：

- `.env.example`
- `SECURITY.md`

这样可以显著降低“项目看上去像 demo”的感觉。

## 7. 风险与取舍

### 风险 1：写得太重，维护成本升高

**取舍：**
先写轻量、准确、可维护版本，不追求第一次就过度完整。

### 风险 2：仓库门面与产品页文案不一致

**取舍：**
直接以 `docs/content/prelaunch-asset-pack.md` 的主叙事为准，减少口径分叉。

## 8. 实施拆解

1. 盘点 README / CONTRIBUTING 的口径偏差
2. 重写 README
3. 更新 CONTRIBUTING
4. 新增 `.env.example` 与 `SECURITY.md`
5. 提交实施记录

## 9. 验收标准

满足以下条件视为完成：

- README 能准确解释 ClawPlay 当前产品形态
- 外部用户能在 5 分钟内找到本地启动方式
- 投稿与 PR 两条贡献路径被明确区分
- 环境变量和安全反馈路径有明确说明

## 10. 上线与回滚

### 上线方式

文档文件直接随代码提交，无数据迁移风险。

### 回滚策略

如果仓库门面文案方向不满意，可直接回退对应文档文件，不影响产品运行。
