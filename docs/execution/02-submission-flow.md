# 02 Submission Flow

- Status: Ready
- Owner: Codex + slicenfer
- Last Updated: 2026-03-08

## 1. 背景

ClawPlay 当前已经具备：

- 中文 Soul 目录站首页与详情页
- 一键安装与原始 `SOUL.md` 查看/下载
- 第一版站内埋点与热门榜基础

但“投稿”能力仍停留在静态说明页：

- `src/app/submit/page.tsx` 只是告诉用户“去 GitHub 提交 PR / Issue”
- 普通用户如果不会 Git、没有 GitHub 使用习惯，投稿门槛很高
- 站点侧没有统一的投稿收集、审核、修订、发布流程
- 当前内容质量、来源和协议判断，都靠人工在仓库里零散处理

如果不补齐投稿闭环，ClawPlay 会卡在一个尴尬阶段：

1. 用户知道这个站可以“换灵魂”，但不知道怎么把自己的灵魂贡献进来。
2. 站长知道该优先收哪些内容，但没有统一队列，也没有状态管理。
3. 即使收到了素材，也缺少“提交 -> 审核 -> 修订 -> 发布 -> 上线”的一条正式链路。

因此，第 2 步要优先解决的不是“社区氛围”，而是**内容供给闭环**。

## 2. 目标

本阶段目标：

1. 让不会 GitHub PR 的普通用户，也能通过网页完成 Soul 投稿。
2. 建立一套最小可用的审核流，把“待审 / 待补充 / 拒绝 / 已发布”这些状态规范起来。
3. 让投稿人能看到自己的投稿状态，而不是投出去就石沉大海。
4. 让站点能在不改代码、不重新发版的前提下，发布通过审核的 Soul。
5. 为后续注册登录、作者主页、社区声誉体系预留升级空间，但不在这一轮一次做满。

## 3. 非目标

本阶段不做：

- 完整的注册 / 登录 / 找回密码系统
- 公开的作者主页
- 公开评论、点赞、收藏、关注
- 自动邮件通知系统
- 自动版权审查或自动翻译来源校验
- 多管理员协同权限系统
- 用户自己直接“发布上线”而不经过审核
- 完整的后台运营面板体系

本阶段只做：**网页投稿 + 审核队列 + 状态回看 + 发布入站**。

## 4. 用户价值 / 业务价值

### 用户价值

- 不用懂 Git，也能把自己调教好的 Soul 发到站里。
- 投稿后能知道当前状态，不会觉得“投了但没人理”。
- 如果内容还不完整，可以收到明确的补充方向，而不是直接被拒。
- 发布通过后，自己的 Soul 能真正进入目录、详情页和安装链路。

### 业务价值

- 把内容增长从“站长手工加文件”升级为“站内持续收集”。
- 降低优质 Soul 流失率，提升社区参与感。
- 把版权、来源、协议、适用模型这些关键信息，在投稿时就结构化收齐。
- 为后续热门榜、精选推荐、投稿转化分析提供数据基础。

## 5. 交付范围

本阶段计划交付：

1. 一个正式的站内投稿页，替代当前静态说明页。
2. 一套结构化投稿表单。
3. 投稿成功后的确认页与“查看投稿状态”入口。
4. 一套内部审核入口与状态流转机制。
5. 一套可直接发布入站的数据模型。
6. 站点数据层支持“现有静态 Soul + 已发布投稿 Soul”的混合读取。
7. 必要的埋点事件，帮助判断投稿转化和审核效率。

可能涉及的范围：

- `src/app/submit/`
- 可能新增 `src/app/submissions/[id]/`
- 可能新增 `src/app/admin/` 下的轻量审核入口
- `src/app/api/`
- `src/lib/souls.ts`
- `src/lib/analytics/`
- 新增投稿相关数据库表

## 6. 方案设计

### 6.1 核心决策 1：先不做公开账号体系，先做“匿名投稿 + 私密管理链接”

这是本阶段最重要的边界。

原因：

1. 账号系统一旦上，就会带来密码、验证、找回、封禁、刷号等大量外围问题。
2. ClawPlay 当前最缺的不是“用户身份体系”，而是“内容进站管道”。
3. 早期站点更需要降低投稿门槛，而不是让用户先注册再操作。
4. 没有邮件服务的前提下，强推账号体系反而会把实现复杂度和维护成本抬得很高。

因此，第一版投稿闭环采用：

- 用户无需注册即可投稿
- 提交成功后生成一个**私密管理链接**
- 用户通过这个链接查看状态、补充信息、提交修订
- 管理员通过内部审核入口处理投稿

这是一种非常典型的早期内容站策略：**先把内容闭环打通，再补身份层。**

### 6.2 核心决策 2：已发布投稿必须能直接进入站点，不依赖代码发布

如果通过审核的投稿还要手动改仓库、手动写 `SOULS` 数组、手动重新部署，那它不是真闭环。

因此本阶段需要明确：

- 审核通过并发布的投稿，要能直接进入站点数据源
- 不要求每次新 Soul 上线都重新发版
- 现有 8 个静态 Soul 可以先继续保留，作为“内置精选 / 历史内容”
- 新投稿走数据库发布路径，与静态内容并存

也就是说，这一步不是简单做一个“表单收集箱”，而是给 ClawPlay 增加一条新的内容生产通道。

### 6.3 核心决策 3：投稿表单必须结构化，不能只收一坨 Markdown

只让用户粘贴一个 `SOUL.md`，短期最省事，但后面问题很多：

- 搜索和列表所需字段不完整
- 预览卡缺少摘要、标签、场景等结构化信息
- 版权和来源说明很容易遗漏
- 审核时不好比对，也不好做补充要求

所以第一版表单要同时收两类信息：

#### A. 展示层元信息

- 标题
- 一句话简介
- 分类
- 标签
- 语气 / 风格
- 适用场景
- 兼容模型
- 预览钩子
- 示例 prompt / response

#### B. 内容与版权信息

- 原始 `SOUL.md`
- 作者名称
- 投稿类型（原创 / 翻译 / 改编）
- 原始来源链接（如适用）
- 协议
- 审核备注 / 投稿补充说明
- 联系方式（GitHub / 邮箱 / 其他）

### 6.4 核心决策 4：审核流程采用“轻后台 + 明确状态机”，不追求一开始就像 CMS

第一版不需要一个完整后台系统，但必须有清晰状态流。

建议状态流如下：

```text
draft (仅服务端临时态，可选)
-> pending_review
-> needs_revision
-> approved
-> published
-> rejected
```

说明：

- `pending_review`：用户已正式提交，等待审核
- `needs_revision`：审核认为可以继续，但信息不全、版权不清或内容需调整
- `approved`：审核通过，但尚未正式发布
- `published`：已进入站点
- `rejected`：不进入站点

其中：

- 用户侧主要能感知：`待审核 / 待补充 / 已通过 / 已发布 / 未采用`
- 管理侧则保留更细的内部状态和值班备注

### 6.5 信息架构

#### 公共侧

1. `/submit`
   - 投稿表单主页面
2. `/submit/success`
   - 提交成功说明页
3. `/submissions/[publicId]`
   - 投稿状态页（需携带私密 token）

#### 管理侧

1. `/admin/login`
   - 轻量管理员登录入口
2. `/admin/submissions`
   - 投稿列表页
3. `/admin/submissions/[id]`
   - 投稿详情与审核操作页

### 6.6 提交流程

投稿人视角：

1. 打开 `/submit`
2. 填写结构化表单
3. 粘贴原始 `SOUL.md`
4. 勾选原创 / 翻译 / 改编声明与版权确认
5. 提交
6. 获得成功页与私密管理链接
7. 后续通过私密链接查看状态或补充内容

管理员视角：

1. 打开审核列表
2. 查看新投稿与状态分布
3. 审核内容完整性、质量、版权来源、站点契合度
4. 给出处理结果：
   - 待补充
   - 拒绝
   - 通过
5. 通过后执行“发布入站”
6. 投稿状态自动变为 `published`

### 6.7 审核标准（第一版）

第一版审核标准建议固定为 4 大类：

1. **可用性**
   - `SOUL.md` 是否可直接用于 OpenClaw
   - 是否存在明显缺段、空内容、乱码、模板未替换
2. **展示完整性**
   - 标题、简介、标签、场景、示例是否足够支撑详情页展示
3. **版权与来源**
   - 原创是否清楚
   - 翻译 / 改编是否标明来源与原作者
   - 协议是否明确
4. **站点契合度**
   - 是否符合 ClawPlay 的中文内容定位
   - 是否存在违法、攻击性、明显侵权或极端低质内容

### 6.8 埋点建议

依赖第 1 步的埋点基础，这一步建议补以下事件：

1. `submission_page_view`
2. `submission_started`
3. `submission_submitted`
4. `submission_success_view`
5. `submission_status_view`
6. `submission_revision_submitted`
7. `admin_submission_reviewed`
8. `admin_submission_published`

这些事件的作用不是炫技，而是回答几个真实问题：

- 打开投稿页的人，有多少真正提交了？
- 表单是不是太长？
- 用户卡在哪一步？
- 投稿转发布的转化率怎样？
- 审核是否积压？

### 6.9 与现有 Soul 数据源的关系

当前站点的 Soul 主要来自：

- 仓库内 Markdown 文件
- `src/lib/souls.ts` 里的静态元数据

本阶段建议采用**双源并存**策略：

1. 当前已上线 Soul 继续保持静态源，不强行迁移
2. 新发布的投稿 Soul 存入数据库
3. `getAllSouls()` / `getSoulBySlug()` 改造成统一读取层
   - 先读静态 Soul
   - 再读数据库中的已发布 Soul
   - 最终合并成站点展示结果

这样既避免“一步到位迁移”的风险，也能让闭环真正上线可用。

## 7. 风险与取舍

### 风险 1：没有账号体系，私密链接丢失后不好找回

处理：

- 第一版接受这个取舍
- 成功页明确提示用户收藏私密链接
- 后续若接入邮件或账号体系，再升级“找回投稿”能力

### 风险 2：开放网页投稿后，垃圾内容和刷表单会增加

处理：

- 第一版增加基础防刷：速率限制、honeypot 字段、内容长度校验
- 不急着先接 CAPTCHA
- 先观察真实压力再追加策略

### 风险 3：动态发布 Soul 与现有静态 Soul 并存，数据模型会出现两套口径

处理：

- 第一版通过统一的数据适配层抹平输出结构
- 不在这一步迁移旧数据
- 等投稿闭环跑顺后，再考虑统一迁移历史 Soul

### 风险 4：审核流做得太重，会反过来拖慢内容增长

处理：

- 第一版保持“单管理员 + 轻状态流”
- 不做复杂协同、不做任务分派
- 只保证清晰、可追踪、能发布

## 9. 验收标准

满足以下条件才算完成：

1. 普通用户无需 GitHub 即可提交一份 Soul 投稿。
2. 投稿提交后会生成可回看的状态入口。
3. 管理员可以看到待审列表，并完成“待补充 / 拒绝 / 通过 / 发布”操作。
4. 一份通过审核的投稿可以在不发版的情况下进入站点目录和详情页。
5. 投稿和审核主流程都有基础埋点可观测。
6. 当前静态 Soul 不受新流程影响，站点原有浏览和安装能力保持正常。

## 10. 上线与回滚

### 上线方式

- 先在本地验证投稿提交、状态页、审核发布与目录展示
- 再部署到 `tm-prod`
- 先低调开放，先通过站内入口收少量真实投稿
- 观察 3 个指标：
  - 提交成功率
  - 审核积压量
  - 已发布内容质量

### 回滚策略

- 投稿相关入口与接口保持模块化
- 如遇垃圾内容压力、状态错乱或动态展示异常，可快速：
  1. 关闭 `/submit` 提交入口
  2. 暂停发布操作
  3. 保留投稿数据，不直接删除
  4. 回退动态 Soul 读取逻辑，仅保留静态 Soul 展示
