# 10 Tag Governance Closed Loop

- Status: Ready
- Depends On: `docs/execution/05-content-governance.md`, `docs/execution/08-filing-wait-priority-roadmap.md`
- Owner: Codex + slicenfer
- Last Updated: 2026-03-09

## 1. 背景

ClawPlay 现在已经有：

- 推荐标签词表
- 标签同义词收敛逻辑
- 投稿端标签输入提示
- 后台内容检查里的标签质量警告

但它还不是一个真正闭环的标签治理系统。

当前缺口在于：

- 用户如果想到一个词表外的新标签，没有明确入口去“提议”
- 审核后台无法对新标签做“归并 / 收录 / 驳回”
- 前台标签和后台判断之间，还没有形成受控流转

这会导致两个问题：

1. 用户容易把新标签直接塞进正式标签字段，数据开始变散。
2. 管理员即使看到了潜在好标签，也没有一个明确的处理动作可以落地。

所以 P3 的目标不是继续加更多提示文案，而是把标签从“建议规则”升级成“实际工作流”。

## 2. 目标

本阶段目标：

1. 给投稿人一个明确的“新标签建议”入口，而不是逼他们乱填正式标签。
2. 让管理员在后台能对标签建议执行 3 类动作：
   - 归并到现有标签
   - 收录为正式标签
   - 驳回标签建议
3. 让发布到前台的仍然是 canonical tags，而不是未经审核的自由标签。
4. 保留标签处理轨迹，后续可以继续扩展词表治理。

## 3. 非目标

本阶段不做：

- 开放用户自由创建公开标签
- 多级标签树
- 标签热度算法
- 自动 AI 标签推荐与自动合并
- 复杂标签运营后台

本轮只做一套够用、清晰、可追溯的标签治理闭环。

## 4. 核心决策

### 决策 1：正式标签与新标签建议分层

投稿端保留两个概念：

- `tags`：正式标签，优先复用推荐词表
- `proposedTags`：用户想提议的新标签

这样可以避免用户为了表达新想法，直接污染正式标签字段。

### 决策 2：后台审核的是“标签建议”，不是整套标签编辑器

第一版后台不做复杂标签 CMS。

管理员只需要围绕“待处理的新标签建议”做 3 个动作：

1. 归并到已有标签
2. 接受为这篇稿件的正式标签
3. 驳回本次建议

这已经能覆盖当前阶段最常见的审核场景。

### 决策 3：收录为正式标签先落到稿件级别，不强做全站动态词表

这一点是本轮的关键取舍。

因为 ClawPlay 当前推荐标签词表仍是代码内静态管理，如果现在就强做“后台动态全站标签词表”，会把范围显著拉大。

因此本轮定义：

- “收录为正式标签”表示该标签进入当前稿件的 canonical tags
- 是否进一步进入全站推荐词表，放到下一轮人工词表扩充中处理

这样既能先完成闭环，也不把架构复杂度一下抬高。

## 5. 交付内容

本阶段至少交付：

1. 投稿端 `proposedTags`
2. 数据层 `proposedTags` 存储
3. 后台标签治理面板
4. 管理员标签处理动作接口
5. 标签治理实施文档

## 6. 完成标志

满足以下条件即可视为本阶段完成：

1. 投稿人可以在表单里单独填写“新标签建议”。
2. 修订稿也能继续看到并编辑这组建议。
3. 后台详情页可直接处理每个标签建议。
4. 管理员处理后，正式标签与待处理建议会立即更新。
5. 发布到前台的仍然是正式标签字段。
