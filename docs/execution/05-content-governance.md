# 05 Content Governance

- Status: In Progress
- Depends On: `docs/execution/02-submission-flow.md`
- Owner: Codex + slicenfer
- Last Updated: 2026-03-08

## 1. 背景

ClawPlay 现在只有少量 Soul，看起来还“差不多都能人工兜住”。

但一旦数量增长到 20、50、100：

- 分类会越来越散
- 标签会越来越碎
- 翻译来源与授权说明容易不一致
- 投稿表单字段会逐渐失去约束力
- 搜索、合集、SEO 和推荐会被脏数据拖垮

所以，内容规范不是“后面再说”的锦上添花，而是平台内容扩张前必须先立住的约束层。

## 2. 目标

本阶段目标：

1. 定义 Soul 元数据的最小必备规范。
2. 建立分类、标签、语气、场景、兼容模型的受控规则。
3. 明确原创 / 翻译 / 改编三类内容的来源与 license 说明要求。
4. 让投稿、审核、发布三段流程都能依照同一套内容规范工作。
5. 为后续搜索、SEO、热门榜、合集页提供稳定的数据地基。

## 3. 非目标

本阶段不做：

- 复杂 CMS 系统
- 自动标签 AI 纠错平台
- 多级内容审核团队流程
- 大而全的 taxonomy 系统
- 内容评分算法

本阶段只做“够严谨、够可执行、能被后台和投稿流程落地”的规范体系。

## 4. 用户价值 / 业务价值

### 用户价值

- Soul 信息更整齐，搜索和筛选更可靠。
- 不同 Soul 的详情页信息结构更一致，阅读成本更低。
- 翻译和改编内容的来源更透明，信任感更强。

### 业务价值

- 平台内容可以规模化扩充，而不是靠人工记忆维持秩序。
- SEO、分享、热门排序、合集编排会更稳定。
- 投稿审核有统一标准，不会每次都重新发明规则。

## 5. 交付范围

本阶段计划交付：

1. Soul metadata 字段规范
2. 分类规范
3. 标签规范
4. 语气 / 场景 / 兼容模型规范
5. 翻译来源与 license 规范
6. 投稿表单与后台审核的字段约束建议
7. 一份可执行的内容风格与命名规则

可能涉及范围：

- `src/lib/souls-types.ts`
- `src/lib/souls.ts`
- `src/lib/submissions/schema.ts`
- `src/lib/submissions/service.ts`
- `src/app/submit/page.tsx`
- `src/app/admin/submissions/[id]/page.tsx`
- 可能新增 `docs/content/` 或 `src/lib/content-rules.ts`

## 6. 方案设计

### 6.1 metadata 最小规范

每个 Soul 至少应具备：

- `title`
- `summary`
- `category`
- `sourceType`
- `tags`
- `tones`
- `useCases`
- `compatibleModels`
- `author`
- `license`
- `previewHook`
- `previewPrompt`
- `previewResponse`
- `intro`
- `features`
- `suggestions`
- `rawSoul`

并建议明确：

- title 长度范围
- summary 长度范围
- tags 数量范围
- preview 三件套的最小质量要求

### 6.2 分类规范

分类必须少而稳。

当前分类：

- `work`
- `dev`
- `learning`
- `creative`
- `translated`

建议原则：

1. 分类用于“一级导航与合集”，不要过多。
2. 分类表达主用途，不表达细碎风格。
3. 若 Soul 为翻译内容，是否同时属于业务分类，需要给出明确规则。

建议补一条决策：

- `translated` 是来源维度，还是独立内容分类？

如果继续保留 `translated` 作为一级分类，就要接受“用途分类”和“来源分类”混在一起；
如果不保留，就应把翻译/改编统一收回 `sourceType`。

这是一个必须先拍板的内容架构问题。

### 6.3 标签规范

标签需要从“随手写”升级为“受控词表 + 小范围开放输入”。

建议：

- 每个 Soul 3~6 个标签
- 标签优先描述能力或场景，不描述空泛褒义词
- 避免重复同义词，例如：
  - `规划` / `计划制定`
  - `编程` / `开发`
- 需要沉淀一份推荐标签词表

### 6.4 语气 / 场景 / 兼容模型规范

这三组字段必须各自承担不同职责：

- `tones`：描述人格或表达气质
- `useCases`：描述适用任务场景
- `compatibleModels`：描述适用模型范围

不能互相混写，否则后面筛选会失真。

### 6.5 原创 / 翻译 / 改编规范

#### 原创

至少要求：

- 作者署名
- license
- 权利声明

#### 翻译

至少要求：

- 原作者
- 原始来源 URL
- 原 license 或授权说明
- 翻译范围说明（全文 / 节选 / 改写）

#### 改编

至少要求：

- 原始来源 URL
- 原作者
- 改动说明
- 当前发布所使用的 license / 说明

### 6.6 命名与风格规范

建议明确：

- Soul title 命名规则
- slug 命名规则
- summary 写法规则
- previewHook 写法规则
- 列表页摘要文风统一要求

### 6.7 后台审核落点

内容规范必须能真正进入审核流程，而不是只停留在文档里。

建议后台审核视角至少有：

- 必填字段完整性检查
- 分类 / 标签是否符合规则
- 来源 / license 是否可公开
- preview 三件套是否有展示价值

## 7. 风险与取舍

### 风险 1：规则定得太死，投稿门槛过高

处理：

- 对投稿人保留适度自由输入
- 对后台审核采用“建议词表 + 人工裁决”模式

### 风险 2：规则定得太松，后续数据还是会乱

处理：

- 优先约束高价值字段：分类、标签、license、来源
- 可逐步扩大约束范围

### 风险 3：来源与授权不清导致内容风险

处理：

- 翻译 / 改编强制补齐来源与说明
- 未明确授权的内容不发布

## 8. 实施拆解

### Task 1：定义内容字段规范

输出：

- metadata 字段说明
- 长度 / 数量 / 质量约束

### Task 2：定义分类与标签规则

输出：

- 分类使用原则
- 推荐标签词表
- 同义词合并策略

### Task 3：定义来源与 license 规范

输出：

- 原创 / 翻译 / 改编要求清单
- 后台审核判定标准

### Task 4：接入投稿与审核流程

输出：

- 投稿端约束建议
- 后台审核提示与校验建议

## 9. 验收标准

满足以下条件视为完成：

1. 新增 Soul 可以按照统一规则填写 metadata。
2. 分类与标签的含义边界明确。
3. 翻译 / 改编内容来源和 license 要求明确。
4. 投稿与后台审核都能引用这套规范。
5. 后续扩容到几十个 Soul 时，不会明显失控。

## 10. 上线与回滚

本阶段更多是规则层和约束层改造。

上线方式：

- 先固化文档与常量
- 再逐步收紧投稿和审核校验

回滚方式：

- 若投稿转化明显下降，可先放宽前台校验
- 但后台审核规范不回滚，保持内容质量底线

## 11. 结果记录

第一阶段已落地：

- `src/lib/content-rules.ts` 内容规则常量与发布评估 helper
- 发布前阻断：翻译 / 改编缺来源、协议占位、预览三件套过弱等内容将不能直接发布
- 后台审核详情新增“内容检查”卡片，明确阻断 / 警告 / 通过项

第二阶段已落地：

- 投稿前台增加实时“投稿预检查”，把后台同一套规则前置为软约束提示
- 根据真实体验反馈，投稿页已调整为“核心必填 + 补充可选展开”，避免把审核视角一次性压给投稿人
- 投稿表单新增推荐长度、最小字数、来源要求和推荐标签 quick pick
- 后台审核工作台新增“发布前摘要”，有阻断项时直接禁用“发布入站”按钮
- 新增 `docs/content/metadata-spec.md` 与 `docs/content/tag-dictionary.md`，沉淀字段规范和标签词表

待继续落地：

- 分类架构是否保留 `translated` 的最终决策
- 旧 Soul 元数据逐步清洗与回填
- 分类页 / 搜索页基于词表的进一步收敛
