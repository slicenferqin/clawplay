## 1. 背景

ClawPlay 当前已经完成了主链路建设：

- 首页与灵魂库可用
- Soul 详情页可用
- 投稿、审核、发布主链路可用
- 安装链路可用
- 基础 SEO 已补齐
- `translated` 分类迁移与来源治理已开始收口

现在域名仍处于备案审核期。

这意味着一个很关键的窗口期出现了：

- 现在不适合盲目继续堆更多页面功能
- 但非常适合把“备案不阻塞、放开即能稳上线”的基础设施和内容治理继续补齐

所以这份文档的目标，不是列一个散乱 TODO，而是给备案等待期定义一份明确的**优先级路线图**。

后续执行，统一按这里的顺序推进，而不是想到什么做什么。

## 2. 目标

本阶段目标：

1. 明确备案等待期最值得做的事项与顺序。
2. 区分“现在必须补”和“可以后面再做”的工作。
3. 为接下来逐项实施提供统一执行顺序和验收口径。
4. 保证域名放开后，ClawPlay 不是“刚好能访问”，而是“能稳上线、能扩内容、能开始传播”。

## 3. 非目标

本阶段不做：

- 重做整站视觉风格
- 上复杂推荐算法
- 上社区互动系统
- 上会员体系或积分体系
- 开放自由创建公开标签
- 做大而全的 CMS 或运营中台

这些都不是当前窗口期最值得投入的内容。

## 4. 用户价值 / 业务价值

### 用户价值

- 域名放开后，访问与安装体验更稳，不容易踩坑。
- 内容筛选、标签和审核结果更可信。
- 新用户首次进入网站时，更容易看懂、选中、安装和投稿。

### 业务价值

- 上线当天不容易出事故。
- 后续 Soul 数量增长时，内容结构不会快速失控。
- 冷启动传播时，不会出现“站点有了，但传播素材和承接页没准备好”的情况。

## 5. 优先级总览

备案等待期建议按下面 6 个优先级推进：

1. **P1：上线工程化**
2. **P2：后台审核效率优化**
3. **P3：标签治理闭环**
4. **P4：发布前内容资产准备**
5. **P5：站点可信度补强**
6. **P6：增长侧准备**

原则：

- 前三项偏基础设施与治理，优先级最高
- 中间两项偏上线承接与稳定性
- 最后一项偏增长放大器，可以在前面基础打稳后再做

## 6. 方案设计

### 6.1 P1：上线工程化

#### 为什么排第一

这是备案放开后最容易出问题、也最值得提前完成的一项。

如果没有工程化检查流程，域名一放开时很容易靠人工点页面排查，效率低且容易漏项。

#### 本阶段目标

把 `06-domain-launch-checklist` 从文档升级为：

- 一份文档
- 一组脚本
- 一条标准命令

#### 交付内容

至少覆盖：

- DNS 解析检查
- 反代 / Nginx 回源检查
- 首页、`/souls`、Soul 详情页 `200`
- `robots.txt`、`sitemap.xml`
- canonical
- OG 图链接
- PM2 进程状态
- 最终 `PASS / WARN / FAIL` 输出

#### 推荐交付形式

- `scripts/ops/domain-smoke-check.mjs`
- `npm run smoke:domain -- --host=https://clawplay.club`
- `docs/execution/06-domain-launch-checklist.md` 回填标准执行方式

#### 完成标志

域名放开当天，不需要临时手工拼命令，直接跑标准命令即可完成基本验收。

### 6.2 P2：后台审核效率优化

#### 为什么排第二

后台现在已经能用，但随着内容增加，审核效率会快速成为瓶颈。

如果不先优化后台，后面新增 Soul 一多，审核速度和判断一致性都会下降。

#### 本阶段目标

让管理员在列表页和详情页里更快完成“先看什么、先处理什么”的判断。

#### 优先交付

- 缺来源稿件优先识别
- “只看缺来源 / 只看阻断项 / 只看待发布”快捷视图
- 列表页更明确的风险摘要
- 必要时支持简单排序策略，例如优先显示翻译 / 改编缺来源稿件

#### 当前状态

这项已经做了第一刀：

- 后台来源筛选已支持
- 列表来源状态已支持
- 详情页来源与授权已前置

#### 下一步重点

把“来源展示”升级成“审核优先级工具”。

### 6.3 P3：标签治理闭环

#### 为什么排第三

标签现在已经有第一轮治理，但还没形成真正的闭环。

如果等投稿量变大后再做，很容易出现：

- 用户输入越来越散
- 新标签没有纳管入口
- 后台无法高效归并

#### 本阶段目标

把“标签规则”升级成“标签生产与审核闭环”。

#### 交付内容

- 投稿端新增 `proposedTags`
- 后台支持：
  - 归并到现有标签
  - 收录为新标签
  - 驳回标签建议
- 标签词表文档继续扩展
- 同义词与推荐词表继续完善

#### 设计原则

- 用户可以提议
- 用户不能直接公开创建正式标签
- 前台只展示 canonical tags

### 6.4 P4：发布前内容资产准备

#### 为什么排第四

备案通过后，真正影响第一波传播效果的，不只是站点能不能打开，还包括你有没有准备好可传播的内容资产。

#### 本阶段目标

把首波对外传播需要的内容物料提前备好。

#### 建议交付

- 首页核心传播文案定稿
- 3-5 个代表 Soul 的推荐卡文案
- 新手安装引导文案
- 投稿引导文案
- 面向不同平台的短文案模板：
  - 微信群
  - 朋友圈
  - 即刻 / X / 小红书等

#### 价值

备案一放开，就可以立刻开始对外传播，而不是临时现场写文案。

### 6.5 P5：站点可信度补强

#### 为什么排第五

这类工作不会立刻带来明显“新功能感”，但它会决定用户是否把这个站点当成一个可靠项目，而不是一个临时 demo。

#### 建议交付

- 健康检查或基础服务可用性检查
- 数据备份 / 回滚方案
- 日志清理与保留策略
- 关于页 / 联系方式 / 投稿规则补强
- 关键空状态、错误状态、回退路径检查

#### 原则

不做重运维平台，只做足够稳、足够能排障的基础补强。

### 6.6 P6：增长侧准备

#### 为什么排第六

增长很重要，但增长应该建立在：

- 站点稳
- 审核顺
- 内容治理不乱
- 传播素材已备好

之后再放大。

#### 建议交付

- “翻译精选”专题页或合集页
- “最适合新手 / 最适合开发 / 最近新增 / 最受欢迎”入口
- 首批种子作者名单与邀请话术
- 初始外部推荐路径设计

#### 原则

先做低成本、高确定性的增长入口，不做复杂增长系统。

## 7. 风险与取舍

### 风险 1：备案等待期继续无序堆功能

如果这段时间想到什么做什么，很容易做出一堆不关键的页面，却把真正重要的上线保障拖后。

**取舍：**
统一按这份优先级路线推进。

### 风险 2：过早做增长，站点底座还不稳

如果先追求传播而不是先补工程化与治理，上线后更容易暴露问题。

**取舍：**
先稳底座，再放大流量。

### 风险 3：标签和审核规则继续半人工化

如果一直靠记忆和手工判断，后面内容增长时会越来越乱。

**取舍：**
优先做审核效率和标签闭环，而不是继续无限靠人工兜底。

## 8. 实施拆解

建议按这个顺序执行：

### Step 1

先完成 **P1 上线工程化**。

### Step 2

再完成 **P2 后台审核效率优化** 的第二刀。

### Step 3

然后完成 **P3 标签治理闭环**。

### Step 4

再整理 **P4 发布前内容资产准备**。

### Step 5

之后补 **P5 站点可信度补强**。

### Step 6

最后再进入 **P6 增长侧准备**。

## 9. 验收标准

满足以下条件，可视为这份路线图已被正确采用：

1. 后续执行文档都按这里的优先级排序推进。
2. 备案放开前，`P1` 至少达到可执行状态。
3. 后台审核不再只是“能用”，而是具备清晰的来源和风险处理路径。
4. 标签不再只靠人工约束，而是进入可持续治理状态。
5. 域名放开后，站点具备立即传播所需的基础内容资产。

## 10. 上线与回滚

这份文档本身不涉及直接上线风险。

它的作用是：

- 统一后续执行顺序
- 减少“凭感觉切任务”的成本
- 让后续每一阶段都有明确上下文和依赖关系

如果后续优先级判断发生变化，应通过新增文档或更新本文件来记录，而不是口头切换。
