# ClawPlay Execution Docs

## 为什么要先写规划文档

ClawPlay 后续迭代统一采用：

1. 先写规划文档
2. 评审确认
3. 再开始实施
4. 实施后补充结果与复盘

这样可以把问题拆成两类：

- **设计问题**：目标不清、边界不清、方案本身不稳
- **执行问题**：方案是对的，但实现漏了、做偏了、验证不够

## 文档命名规则

统一放在 `docs/execution/` 下。

建议命名：

```text
NN-topic-name.md
```

示例：

```text
01-analytics-and-hotlist.md
02-submission-flow.md
03-seo-and-sharing.md
04-domain-launch-checklist.md
```

## 建议采用双层文档结构

每个主题建议至少有两层文档：

1. **规划文档**
   - 回答“为什么做、做什么、不做什么、如何验收”
2. **实施细化文档**
   - 回答“具体怎么落、改哪些文件、接口怎么定、数据怎么存、按什么顺序做”

推荐命名：

```text
01-analytics-and-hotlist.md
01-analytics-and-hotlist-implementation.md
```

这样后面复盘时，就能更清楚地区分：

- 是规划阶段没设计好
- 还是实施阶段没按设计执行到位

## 每份规划文档的固定结构

### 1. 背景

说明为什么现在要做这一步。

### 2. 目标

定义本阶段真正要达成的结果。

### 3. 非目标

明确这一步不做什么，防止范围蔓延。

### 4. 用户价值 / 业务价值

说明这一步完成后，对用户和站点分别带来什么价值。

### 5. 交付范围

列出本次会新增、修改、输出的页面、接口、数据结构、文档等。

### 6. 方案设计

说明关键设计决策，包括：

- 信息结构
- 数据模型
- 事件流 / 状态流
- 页面交互
- 技术选型

### 7. 风险与取舍

说明：

- 方案可能出什么问题
- 为什么选择这个方案，而不是其它方案

### 8. 实施拆解

把执行拆成若干可验证的小任务。

### 9. 验收标准

用可以检查的方式定义“做完”的标准。

### 10. 上线与回滚

说明上线方式、依赖条件和回滚策略。

### 11. 结果记录

实施完成后补：

- 最终提交
- 实际偏差
- 遇到的问题
- 后续优化项

## 每份实施细化文档建议补充的内容

实施细化文档不必完全照搬规划结构，但至少应补清这几类信息：

1. **关键决策收口**
   - 比如到底选 SQLite 还是 JSONL
2. **模块与文件落点**
   - 新增哪些文件，改哪些文件
3. **接口契约**
   - 请求、响应、字段约束
4. **数据结构 / 表结构**
   - 字段、索引、扩展位
5. **接入点清单**
   - 哪个按钮、哪个页面、哪个组件承担什么职责
6. **实施顺序**
   - 哪一步先做，哪一步后做
7. **验证路径**
   - 开发完成后最少要跑哪些检查

## 建议的状态字段

每份文档顶部建议带状态：

- `Draft`
- `Ready`
- `In Progress`
- `Done`

## 复盘原则

如果结果不理想，要优先判断：

1. 是不是规划阶段就没定义清楚
2. 是不是边界没有锁住
3. 是不是实现阶段没有按规划落地
4. 是不是验收标准不够具体

先找“哪一层出错”，再改，不要直接靠感觉继续堆改动。
