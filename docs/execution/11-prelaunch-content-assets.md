# 11 Prelaunch Content Assets

## 1. 背景

ClawPlay 的主链路已经基本打通：

- 首页、灵魂库、详情页可用
- 投稿、审核、发布链路可用
- 安装命令、原文查看、复制与下载可用
- SEO、分享图、上线 smoke 检查已补齐
- 标签治理闭环已经建立

备案等待期的下一步，不是继续堆功能，而是把“上线第一波传播需要的内容资产”提前准备好。

因为真实的上线效果，取决于两件事：

1. 站点能不能打开
2. 用户进来后，能不能快速理解、快速挑选、快速安装、快速愿意分享

现在如果不把内容资产先准备好，等域名放开后就会出现：

- 首页能打开，但主文案还不够统一
- 推荐 Soul 没有成体系的对外介绍口径
- 安装与投稿说明在不同页面里口径不一致
- 对外传播时临时写群发文案、朋友圈文案、Jike/X 文案，效率低且风格发散

所以 P4 的目标，是把 ClawPlay 从“站点已经能用”，再推进到“上线就能讲清楚、能传播、能承接”。

## 2. 目标

本阶段目标：

1. 统一 ClawPlay 对外的一句话定位与扩展说法。
2. 产出首批可直接使用的传播内容资产，而不是停留在想法层。
3. 让首页、安装页、投稿页、社交传播文案使用同一套口径。
4. 为域名放开后的首波传播准备即取即用的文案包。

## 3. 非目标

本阶段不做：

- 重做整站视觉系统
- 大改首页交互结构
- 做品牌故事长文或 About 页面完整版
- 上复杂运营后台或内容排期系统
- 针对每个平台做完全不同的一套品牌语气

本阶段只做“上线前最值钱、最能直接拿来用”的那一层内容资产。

## 4. 用户价值 / 业务价值

### 用户价值

- 新用户更快理解 ClawPlay 是什么。
- 新用户更容易知道“我该装哪个 Soul”。
- 新用户更容易完成第一次安装与第一次投稿。
- 用户转发给别人时，能直接复用一套清晰说法。

### 业务价值

- 域名放开后，传播与承接不脱节。
- 首页、安装页、投稿页口径统一，减少认知摩擦。
- 首波传播更像一个成型项目，而不是“功能有了，但说不明白”。
- 后续新增 Soul、写帖子、做合集时，有统一文案基座可复用。

## 5. 交付范围

本阶段至少交付三类内容：

1. **定位与主叙事**
   - 一句话定位
   - 首页主标题 / 副标题候选
   - 对外介绍的长短版说法

2. **首批内容资产**
   - 3-5 个代表 Soul 推荐文案
   - 安装引导文案
   - 投稿引导文案
   - 常见问题的标准回答口径

3. **传播模板**
   - 微信群短文案
   - 朋友圈文案
   - 即刻 / X / 小红书等短文案模板
   - 发布当天的“首发贴”结构建议

建议落地文件：

- `docs/content/prelaunch-asset-pack.md`
- 如需后续接入页面，再补充到首页、安装页、投稿页组件文案中

## 6. 方案设计

### 6.1 内容主张

ClawPlay 当前最应该强调的，不是“我们做了很多功能”，而是三件用户立刻能感知的价值：

1. **可浏览**：不再靠群聊或零散链接捡 Soul
2. **可比较**：先看预览、标签、简介，再决定要不要装
3. **可安装**：复制命令即可落地，安装门槛低

因此对外主张应围绕：

> 把零散的 `SOUL.md` 整理成一个可浏览、可比较、可安装的中文灵魂库。

而不是强调：

- “我们是社区”
- “我们是交易平台”
- “我们是生态基础设施”

这些说法对当前阶段过重，也不利于首波传播。

### 6.2 语气策略

建议统一为：

- 中文、清晰、克制
- 少黑话，少自嗨
- 有一点角色感，但不油腻
- 以“帮用户更快选到适合自己的 Soul”为中心

不建议采用：

- 过于技术白皮书式的冷硬表述
- 过于社区口号式的亢奋语气
- 太多“革命”“生态”“下一代”式夸张表达

### 6.3 首页与传播文案的关系

首页文案不应与社交传播文案完全一致，但必须来自同一叙事母体。

推荐结构：

- 首页负责把价值讲清楚
- 推荐卡负责把个体 Soul 讲生动
- 安装页负责把行动成本降下来
- 投稿页负责把参与门槛降下来
- 社交文案负责把“为什么值得点开”讲出来

### 6.4 首批推荐 Soul 的选择原则

首批只选 5 个左右，覆盖不同使用场景，而不是平均分配：

1. 决策 / 规划：`edict-counselor`
2. 开发 / 审查：`code-reviewer`
3. 日常开发陪跑：`grumpy-wang`
4. 创意与角色感：`catgirl-nova`
5. 架构与系统设计：`architect`

这样可以保证首波传播时，用户一眼就能看到“这不是 5 个差不多的 Soul”。

## 7. 风险与取舍

### 风险 1：文案写得太满、太长

如果每段都想讲完整，首页会变重，传播文案也不易转发。

**取舍：**
先保留短版、可复用版本；长版说明只作为补充。

### 风险 2：每个平台都想定制，导致维护成本变高

**取舍：**
先统一一套主叙事，再按渠道做轻量改写，不做完全独立口径。

### 风险 3：推荐 Soul 选择过多，反而削弱主推效果

**取舍：**
先锁 5 个代表 Soul，后续再滚动更新，不追求一开始就全覆盖。

## 8. 实施拆解

1. 盘点当前首页、安装页、投稿页现有文案与口径
2. 定义统一的一句话定位和长短版介绍
3. 产出首批推荐 Soul 介绍文案
4. 产出安装引导、投稿引导与 FAQ 口径
5. 产出社交传播模板
6. 后续再按需要同步回页面组件

## 9. 验收标准

满足以下条件，视为本阶段第一刀完成：

- 有一份统一的预发布内容资产包文档
- 至少包含 5 个代表 Soul 的推荐文案
- 至少包含安装、投稿、分享三类模板文案
- 首页、安装、投稿三条主链路的表达口径可映射到同一套主叙事
- 任何一个对外传播动作，不需要再从零写第一版文案

## 10. 上线与回滚

### 上线方式

本阶段以文档资产为主，不涉及高风险数据迁移。

如果后续接入页面文案：

- 先按资产包回填到对应页面
- 再走一次 UI 与文案回归

### 回滚策略

如果后续页面文案改动效果不好：

- 页面可以直接回退到旧文案
- 资产包文档保留，继续作为内部运营素材使用
