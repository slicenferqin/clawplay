# 01 Analytics and Hotlist

- Status: Draft
- Owner: Codex + slicenfer
- Last Updated: 2026-03-08

## 1. 背景

ClawPlay 首页已经开始使用“动态推荐 soul”的 Hero 展示，但当前的 5 个 soul 仍然是人工挑选，不是基于真实用户行为得出的结果。

如果继续在没有数据的前提下使用“热门”“推荐”这类表达，会出现两个问题：

1. 首页推荐缺少客观依据。
2. 后续难以判断哪些 soul 真正有吸引力，哪些页面交互真正有效。

因此，下一步要优先补齐**轻量埋点 + 热门榜基础能力**。

## 2. 目标

本阶段目标：

1. 建立一套最小可用的站内行为埋点方案。
2. 让首页 Hero 的“推荐 soul”具备可被数据驱动替换的基础。
3. 形成第一版“热门排序规则”，即使规则很简单，也要可解释、可追溯。
4. 为后续投稿、SEO、转化优化提供基础数据支撑。

## 3. 非目标

本阶段不做：

- 第三方复杂分析平台接入（如 GA4、Mixpanel、PostHog Cloud）
- 用户级画像系统
- 登录用户行为归因
- A/B 测试平台
- 实时看板系统
- 复杂推荐算法
- 个性化推荐

本阶段只做“够用、稳定、可落地”的 MVP 数据能力。

## 4. 用户价值 / 业务价值

### 用户价值

- 首页推荐更贴近真实受欢迎内容。
- 详情页和安装入口会围绕真实使用行为持续优化。

### 业务价值

- 能判断哪些 soul 真正有吸引力。
- 能判断首页 Hero、详情页、安装入口的转化效果。
- 为后续“热门榜”“精选推荐”“首页排序”提供依据。

## 5. 交付范围

本阶段计划交付：

1. 一套事件模型文档。
2. 站内埋点基础设施。
3. 关键事件上报实现。
4. 一份简单的热门计算规则文档。
5. 第一版热门 soul 数据读取/输出能力。
6. 必要的管理脚本或数据读取接口。

可能涉及的代码范围：

- `src/app/`
- `src/components/`
- `src/lib/`
- 可能新增 `src/lib/analytics.ts`
- 可能新增简单的数据落盘或存储层

## 6. 方案设计

### 6.1 事件模型

第一版只采集这些事件：

1. `home_hero_view`
   - 首页 Hero 曝光
2. `hero_soul_preview_click`
   - Hero 右卡点击“看预览”
3. `hero_install_copy`
   - Hero 右卡点击“复制安装”
4. `soul_detail_view`
   - Soul 详情页访问
5. `detail_install_copy`
   - 详情页复制安装命令
6. `detail_raw_open`
   - 打开原始 SOUL
7. `detail_raw_download`
   - 下载 `SOUL.md`
8. `detail_raw_copy`
   - 复制原文

### 6.2 事件字段

每条事件至少包含：

- `event_name`
- `slug`（如有）
- `source`（hero / detail / list）
- `timestamp`
- `session_id`
- `path`
- `user_agent`（服务端可选采集）

### 6.3 存储策略

第一版建议使用**自托管轻量存储**，优先顺序：

1. SQLite
2. JSONL 落盘（仅极简过渡方案）

不建议第一版直接接第三方 SaaS，因为：

- 增加外部依赖
- 域名尚未完成备案切换
- 当前优先级是快速形成闭环，而不是平台化

### 6.4 热门排序规则（第一版）

建议用简单加权分：

```text
hot_score =
  detail_view * 1
+ hero_preview_click * 2
+ install_copy * 3
+ raw_download * 2
+ raw_copy * 1
```

说明：

- 详情浏览是基础兴趣
- 复制安装是高意图行为，权重更高
- 下载 raw 说明用户有较强试用意愿

### 6.5 首页 Hero 的使用方式

第一版不直接自动替换 Hero 的 5 个 soul，而是分两阶段：

#### Phase A

- 先埋点
- 先算分
- 先能看到热门排序结果

#### Phase B

- 再决定是否自动使用热门前 5
- 或继续保留人工精选 + 数据参考的混合策略

这样可以避免数据样本不足时，首页推荐出现大幅波动。

## 7. 风险与取舍

### 风险 1：事件太少，热门结果不稳定

处理：

- 第一版允许人工精选优先
- 热门榜先作为“内部参考数据”使用

### 风险 2：埋点过重，影响性能

处理：

- 只埋关键事件
- 上报异步化
- 不阻塞主交互

### 风险 3：事件设计不清，后面难用

处理：

- 第一版只采集真正会用于决策的行为
- 保持事件命名和字段统一

### 风险 4：没有登录，去重不完美

处理：

- 第一版接受 session 级别统计
- 不追求用户级精确归因

## 8. 实施拆解

### Task 1：定义事件模型与存储结构

输出：

- 事件名清单
- 字段定义
- 数据表 / 文件结构

### Task 2：实现埋点写入能力

输出：

- 前端触发方法
- 服务端写入入口
- 基础存储能力

### Task 3：接入关键页面与按钮

输出：

- 首页 Hero 埋点
- 详情页埋点
- 下载 / 复制 / 原文查看埋点

### Task 4：实现热门计算脚本或读取接口

输出：

- 简单排序逻辑
- 热门结果读取能力

### Task 5：文档与校验

输出：

- 埋点说明
- 本地验证方式
- 上线检查项

## 9. 验收标准

满足以下条件才算完成：

1. 关键事件都能被触发并写入存储。
2. 每个事件都有统一字段结构。
3. 能基于事件数据生成一份 soul 热门排序结果。
4. 首页与详情页核心交互不受埋点阻塞。
5. 文档里能清楚说明“为什么这样设计”。

## 10. 上线与回滚

### 上线方式

- 先在本地验证事件触发
- 再部署到 `tm-prod`
- 先观察事件是否正常积累

### 回滚策略

- 埋点逻辑通过独立模块接入
- 如发现异常，可快速关闭上报入口或回退相关提交
