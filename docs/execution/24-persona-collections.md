# 24 Persona Collections PRD

- Status: Done

- Owner: Codex + slicenfer
- Last Updated: 2026-03-10

## 1. 背景

ClawPlay 已经有一套可用的“入口型合集”：

- 新手首选
- 开发首选
- 当前热门
- 最近新增

这套合集解决的是：**第一次来时先从哪里看**。

但当产品叙事已经从“下载一个 `SOUL.md`”升级到“理解并选择一种人格”之后，只靠入口型合集还不够。

用户下一步会自然提出新的问题：

- 我想找更像“谋士”的 Soul，有没有一组？
- 我想找更温润、更有陪伴感的 Soul，有没有一组？
- 我能接受被直说，想找更毒舌、更锋利的 Soul，有没有一组？
- 我不想要中性工具，我想找角色感强的人格，有没有一组？

这就是 `N5` 要解决的问题：

**让合集从“用途入口”进化到“人格专题”。**

## 2. 目标

本期只做第一版人格专题合集，目标不是做推荐算法，而是先提供一组**可理解、可传播、可比较**的人格专题落点。

本期目标：

1. 保留现有入口型合集
2. 新增一组人格型合集
3. 合集总览页同时承载这两类合集
4. 每个专题页保持可独立传播、可复制文案、可分享 OG 图

## 3. 非目标

这期先不做：

- 个性化推荐算法
- 基于用户行为的自动人格匹配
- AI 自动生成合集说明文案
- 独立的人格筛选器与复杂排序器
- 细到“暴躁系 / 温润系 / 粘人系”全量拆分

原因很直接：

- 当前 Soul 数量还少，过细拆分会造成很多专题过薄
- 本期优先做稳定、清晰、对传播友好的第一版人格专题

## 4. 第一版人格专题范围

结合当前 8 个 Soul 的 bootstrap 人格分析结果，第一版先落这 4 组：

### 4.1 谋士系

关键词：规划、决策、边界、结构、秩序。

典型特征：

- 结构感高
- 边界感高
- 主动性偏高
- 更适合复杂问题拆解，而不是纯陪聊

当前适合纳入的代表：

- 三省六部·御用谋士
- 软件架构师
- 敏捷项目经理

### 4.2 温润陪伴系

关键词：温度感、陪伴感、对话柔和、容易被接住。

典型特征：

- 温度感高
- 锋利度低或中低
- 更适合陪伴、聊天、解释和轻量引导

当前适合纳入的代表：

- 猫娘 Nova
- 苏格拉底导师
- 海盗船长

### 4.3 毒舌直给系

关键词：锋利、指出问题、不绕弯、质量收口。

典型特征：

- 锋利度高
- 边界感高
- 结构感高
- 更适合 review、调试、挑错和快速收口

当前适合纳入的代表：

- 暴躁老王
- 代码审查官
- 软件架构师

### 4.4 角色感拉满

关键词：角色氛围鲜明、人格可感知、互动记忆点强。

典型特征：

- 角色感高
- 更容易被用户感知到“它是谁”
- 更适合拿来比较人格气质，而不是只比较工具能力

当前适合纳入的代表：

- 猫娘 Nova
- 海盗船长
- 三省六部·御用谋士

## 5. 产品原则

### 5.1 入口型合集与人格型合集并存

入口型合集回答：

- 第一次来先从哪组看
- 最近大家在看什么
- 我想看开发向入口在哪里

人格型合集回答：

- 我更喜欢哪种气质
- 我更适合哪种人格表达方式
- 我是否愿意把这类灵魂导入自己的龙虾

两类合集不是替代关系，而是前后衔接关系。

### 5.2 人格专题允许重叠

同一个 Soul 可以同时出现在多个人格专题里。

例如：

- `三省六部·御用谋士` 可以同时属于“谋士系”和“角色感拉满”
- `猫娘 Nova` 可以同时属于“温润陪伴系”和“角色感拉满”
- `软件架构师` 可以同时属于“谋士系”和“毒舌直给系”边缘区域

这是正常的，因为 Soul 本来就不是单标签对象，而是人格复合体。

### 5.3 当前不追求极细分

像“暴躁系”“温润系”“陪伴系”“毒舌系”这些词，长期都值得做。

但以当前 8 个 Soul 的量级，更稳妥的第一版是先用 4 个覆盖面更强的人格专题，把专题页做厚、把传播文案做顺。

## 6. 用户路径

### 路径 A：新用户

1. 进入 `/collections`
2. 先看入口型合集
3. 再切到人格型合集
4. 点进某个专题页
5. 继续进入 Soul 详情页做最终判断

### 路径 B：已有偏好的用户

1. 直接进入 `/collections`
2. 找到自己喜欢的人格专题
3. 先看一组气质相近的 Soul
4. 对比详情页的人格结构与原始 `SOUL.md`
5. 决定是否导入

## 7. 页面要求

### 7.1 `/collections`

需要做的事：

- 明确分成“入口型合集”和“人格专题”两块
- 总览卡片同时展示两类合集
- 保持阅读稳定，避免信息拥挤和布局跳动
- 让用户一眼看懂：这是“先按入口看”还是“先按人格看”

### 7.2 `/collections/[key]`

专题详情页继续沿用现有结构：

1. 一句话专题定位
2. 为什么先看这组
3. 适合谁
4. 推荐 Soul
5. 分享这组

但文案要更偏“人格比较”，而不是普通列表说明。

## 8. 数据与规则要求

第一版不新增数据库表。

直接基于现有：

- `SoulDocument`
- `personaAnalysis.publicScores`
- `buildPersonaProfile(...)`

动态生成人格专题。

推荐规则：

- 使用公开人格维度分数做基础筛选
- 再结合 archetype、标题、摘要、标签、tones 做轻量修正
- 允许少量 curated slug 优先级，保证当前 8 个 Soul 的落点更稳

## 9. 验收标准

满足以下条件即可视为 N5 第一版完成：

1. `/collections` 可同时展示入口型合集与人格专题
2. 至少 4 个人格专题可访问
3. 每个人格专题页都具备独立 metadata、OG 图与分享文案
4. 人格专题的内容不是纯手写硬编码 slug，而是带规则生成
5. `npm run build` 通过
6. README 和版本文档同步记录 N5 落地情况

## 10. 风险与取舍

### 风险 1：Soul 太少，专题会过薄

取舍：

- 第一版只做 4 组
- 允许 Soul 重叠归属
- 用“更宽的人格主题”替代过细标签

### 风险 2：人格专题和入口型合集混淆

取舍：

- 总览页强制分组
- 每组单独写清楚用途
- 专题文案明确这是“按人格气质挑”

### 风险 3：规则太死，后续扩展困难

取舍：

- 先做轻规则 + 轻 curated 的混合策略
- 后续 Soul 数量上来后，再升级成独立权重配置或后台治理

## 11. 后续演进

当 Soul 数量继续增长后，下一步再考虑：

- 细分到“暴躁系 / 温润系 / 陪伴系 / 毒舌系”
- 支持专题排序理由可视化
- 按人格维度做筛选器
- 做“人格对比视图”

- 已保留入口型合集，并在总览页做分组展示
