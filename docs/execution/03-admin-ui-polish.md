## 1. 背景

投稿闭环 MVP 已经跑通，但管理员后台仍停留在“能用”的阶段：

1. 列表页缺少状态概览，页面信息层级偏散。
2. 筛选、检索、表格之间的节奏不稳定，既有空白浪费，也有组件贴得很紧。
3. 详情页的“审核操作 / 投稿信息 / 状态记录 / 修订历史”层次不够清晰。
4. 当管理员连续处理多条投稿时，页面缺乏一个稳定、成熟的后台观感。

因此，需要补一轮专门面向管理员工作流的 UI 精修。

## 2. 目标

本阶段目标：

1. 让后台列表页形成“概览 → 筛选 → 明细”的稳定阅读顺序。
2. 让详情页形成“审核工作台 → 内容判断 → 侧栏资料”的稳定操作顺序。
3. 统一后台卡片、工具栏、时间线、按钮的间距与层级。
4. 在不引入复杂后台框架的前提下，把当前审核后台提升到可长期使用的质感。

## 3. 非目标

本阶段不做：

- RBAC 权限系统
- 多管理员协作
- 审核备注富文本
- 批量审核
- 拖拽式后台布局器
- 复杂图表或实时监控看板

## 4. 设计原则

### 4.1 队列优先

管理员先关心“现在积压多少、哪类状态最多、我该先处理什么”，因此列表页必须优先呈现状态概览。

### 4.2 工作台优先

详情页不是给普通用户看的详情页，而是给审核者做决策的工作台，所以“审核动作区”必须成为主视觉锚点。

### 4.3 降低信息拥挤感

采用更明确的卡片分区、更稳定的 padding、更清晰的子标题层级，避免内容堆叠在一起。

### 4.4 尽量在列表页完成粗筛

标题、摘要、编号、分类、时间应该在列表页就足够帮助管理员判断是否值得点开。

### 4.5 稳健、克制、可读

继续沿用站点既有的温润浅色基调，不切换到过暗或过强风格，优先保证长时间阅读和处理任务的舒适性。

## 5. 交付范围

计划交付：

1. 列表页状态概览区
2. 独立工具栏卡片（筛选 + 搜索）
3. 更成熟的投稿表格卡片
4. 详情页顶部摘要区
5. 更清晰的审核工作台布局
6. 更舒展的侧栏信息卡与时间线样式
7. 一轮后台样式统一调整

涉及文件：

- `src/app/admin/submissions/page.tsx`
- `src/app/admin/submissions/[id]/page.tsx`
- `src/components/admin-decision-form.tsx`

- 如有必要，少量服务层统计辅助

## 6. 方案设计

### 6.1 列表页

列表页拆成 4 层：

1. 顶部 Hero：标题 + 说明 + 当前队列摘要
2. 状态概览：展示全部 / 待审核 / 待补充 / 已通过 / 已发布 / 未采用
3. 工具栏卡片：筛选 pills + 搜索
4. 表格卡片：标题、摘要、编号、分类、时间、操作

### 6.2 详情页

详情页拆成 3 层：

1. 顶部摘要：标题、状态、基础元信息
2. 左主区：审核工作台、展示信息、预览内容、原始 SOUL
3. 右侧栏：版权与联系、状态记录、修订历史

### 6.3 审核操作区

把操作分成两段：

1. 审核流转：待补充 / 拒绝 / 通过
2. 发布动作：发布入站

这样可减少“状态流转”和“最终发布”混在一起带来的误操作。

### 6.4 视觉规则

- 提高后台触达尺寸，保持主按钮和 pills 至少 44px 高
- 使用卡片化容器明确分区
- 时间线改为更易扫读的垂直轨道样式
- 表格增强 hover 与行内摘要信息
- 控制长文本区域高度，避免内容把整体页面撑得跳动

## 7. 风险与取舍

### 风险 1：后台页面过度设计

处理：

- 不引入复杂组件库
- 不做炫技动画
- 只围绕审核效率和信息清晰度优化

### 风险 2：移动端过度适配造成复杂度上升

处理：

- 以桌面审核体验为主
- 在窄屏下保证可读和不溢出即可

### 风险 3：样式改动影响前台页面

处理：

- 尽量使用 admin 专属类名
- 避免大面积改写全局基础组件行为
