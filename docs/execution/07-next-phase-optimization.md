## 1. 背景

ClawPlay 当前主链路已经基本可用：浏览、搜索、详情、投稿、后台审核、安装、基础 SEO 都已经打通。

接下来剩下的，不再是“今天不上就不能用”的问题，而是会直接影响第二阶段扩张质量的问题：

1. `translated` 这个分类是否继续保留
2. 标签词表是否继续扩充，以及如何扩充
3. 已有 Soul 数据是否要做更细的二次清洗
4. 域名放开后的上线检查清单是否要工程化

这些问题如果现在不收口，后面数据一多，改动成本会明显放大。

## 2. 目标

本阶段目标不是再做新功能，而是把内容结构、标签策略和上线保障补到“可规模化”的状态。

具体目标：

1. 明确 `translated` 在站点 IA 中的最终定位。
2. 建立“受控词表 + 提议机制”的标签治理方案。
3. 完成对现有静态 Soul 的第二轮质量清洗标准。
4. 把域名放开后的上线检查从文档升级为可执行脚本和固定流程。

## 3. 非目标

本阶段不做：

- 引入复杂 CMS
- 上线自动 AI 内容审核系统
- 做多级标签树或庞大 taxonomy
- 把投稿系统变成开放编辑社区
- 引入用户自定义公开标签的完全自由模式

## 4. 用户价值 / 业务价值

### 用户价值

- 浏览路径更清晰，不会混淆“内容用途”和“内容来源”。
- 标签更稳定，搜索和筛选命中率更高。
- 新域名放开后，站点可访问性、分享、收录和故障排查更稳。

### 业务价值

- 后续从 8 个 Soul 扩到 30、50、100 个时，内容结构不会失控。
- 投稿审核成本更低，后台决策更一致。
- 上线和换域名不再依赖人工记忆，减少线上事故。

1. 分类与来源维度解耦方案
2. 标签词表扩充与提议机制方案

## 6. 方案设计

### 6.1 `translated` 的最终定位

这是当前最重要的结构决策。

#### 结论

**不建议把 `translated` 长期保留为公开一级分类。**

更合理的结构是：

- `category` 只表达“主要用途”
  - `work`
  - `dev`
  - `learning`
  - `creative`
- `sourceType` 表达“内容来源”
  - `原创`
  - `翻译`
  - `改编`
- 如确实需要强调翻译内容，可额外提供一个“翻译精选”合集或筛选入口，而不是把它混进主分类。

#### 原因

因为 `translated` 本质上不是用途，而是来源。

如果继续保留 `translated` 为一级分类，会产生三个问题：

1. 同一个 Soul 会同时具有“用途属性”和“来源属性”，但分类位只能表达一个维度。
2. 导航会越来越别扭：`开发专家`、`学习伙伴`、`翻译精选` 不是同一层级概念。
3. 后续 SEO、列表、推荐、统计都会混淆口径。

#### 推荐迁移策略

不要硬切，采用两阶段迁移：

**阶段 A：前台 IA 先解耦，数据先兼容**

- 列表和详情页开始强化 `sourceType` 展示
- 新增“翻译 / 原创 / 改编”筛选能力
- UI 不再把 `translated` 当长期主导航重点
- 老数据允许暂时保留 `translated` 值，避免一次性大迁移

**阶段 B：数据正式收口**

- 把 `translated` 分类内容重新归入真正用途分类
- `translated` 从类型定义中标记为 deprecated，最终移除
- 如有必要，增加“翻译精选”专题页或 collection 字段承接

#### 最终建议

**产品层：不保留。兼容层：短期保留，逐步下线。**

### 6.2 标签词表的扩充方案

当前词表已经有一轮收敛，但规模还偏小，离“可持续扩张”还有一步。

#### 结论

标签不应完全自由生长，而应采用：

**受控词表 + 同义词归并 + 用户提议标签**

#### 目标结构

把标签分成三层：

1. **Canonical Tags**
   - 公开展示、搜索、筛选、SEO 使用的正式标签
   - 由平台维护
2. **Alias / Synonyms**
   - 用户输入、历史旧标签、同义词映射
   - 不直接公开展示，进入归一化流程
3. **Proposed Tags**
   - 用户投稿时可补充的候选标签
   - 不自动成为正式标签，需要后台确认

#### 扩充原则

下一轮建议把正式标签从当前规模扩到 **35–45 个**，但不要一次冲到 80+。

推荐按三组扩：

1. **能力类**
   - 规划、策略、开发、调试、代码审查、测试、架构、写作、翻译、复盘
2. **场景类**
   - 需求拆解、技术讲解、文档整理、学习辅导、角色扮演、思辨/提问
3. **工作流类**
   - 项目管理、系统设计、方案评审、信息整合、问题定位、决策支持

#### 不建议做的事

- 不建议把情绪词、褒义词、梗词直接做成正式标签
- 不建议把“翻译”“原创”这类来源属性混进功能标签
- 不建议让标签承载分类职责

### 6.3 标签是否开放给用户自己创建

#### 结论

**不建议开放为“用户创建即公开生效”。**

但也**不建议完全封死**。

最佳策略是：

**前台可提议，后台决定是否纳入词表。**

#### 为什么不能完全开放

如果完全开放，几周内就会出现：

- 同义词横飞：`开发` / `编程` / `coding`
- 近义词碎片化：`思考` / `思辨` / `提问引导`
- 情绪化标签泛滥：`超强` / `炸裂` / `有趣`
- SEO 噪音增加
- 搜索和筛选变得不可信

这对早期内容平台是非常伤的。

#### 为什么也不该完全关闭

如果完全不允许用户补充，会丢掉两类高价值信息：

1. 词表还没覆盖到的新场景
2. 社区自发形成的新叫法

所以更好的平衡是：

- 投稿时允许用户额外填写 `0-3` 个“补充标签建议”
- 系统提示这些标签会进入审核，不会直接公开展示
- 后台提供三个动作：
  - 归并到现有标签
  - 作为新标签收录
  - 驳回

#### 实施建议

投稿端新增：

- `proposedTags: string[]`
- 文案：`可补充 0-3 个你认为最贴切的标签建议，发布前会由平台统一整理。`

后台审核新增：

- 现有标签建议映射
- 新标签批准按钮
- 驳回原因记录（可选）

公开前台仍然只展示正式标签。

### 6.4 存量数据二次清洗方案

第一轮清洗已经完成“能看、能用、基本统一”，第二轮应改成“有标准地清洗”。

#### 二次清洗目标

针对当前 8 个静态 Soul，进一步校正以下内容：

1. 标题与 summary 的风格一致性
2. `previewHook` 是否足够像“预览卡片钩子”而不是说明文
3. `previewPrompt` / `previewResponse` 是否足够短、够像真实互动
4. 标签是否仍有边缘词、重复词、弱词
5. `author` / `license` / `sourceUrl` / `rightsStatement` 是否完全一致
6. `rawSoul` 中是否还有格式残缺、编码异常、段落不齐

#### 建议做法

先做一份审计清单，而不是凭感觉逐个翻文件。

建议产出一个内容审计结果表，按 Soul 列出：

- metadata 合规性
- 标签合规性
- 预览质量
- 来源与授权完整性
- 原文排版质量

然后集中修一次，而不是零散补。

#### 推荐落点

- 增加内容审计脚本或只读校验逻辑
- 输出控制台报告或 markdown 报告
- 人工复核后统一改 `src/lib/souls.ts` 和 `translated/*.md`

### 6.5 域名放开后的上线检查工程化

当前 `06-domain-launch-checklist` 只有文档，没有工程化执行手段。

#### 结论

建议把它升级成：

**一份文档 + 一组脚本 + 一条固定上线命令**

#### 目标

做到域名可访问后，不靠人工记忆，而是通过命令完成基本验收。

#### 最小工程化范围

至少覆盖以下检查：

1. DNS 是否已解析到目标 IP
2. Nginx / 反代是否正确回源到 `3010`
3. 首页 `200`
4. 关键列表页 `200`
5. 随机一个 Soul 详情页 `200`
6. `robots.txt` 正常
7. `sitemap.xml` 正常
8. canonical 正确
9. OG 图链接可访问
10. 登录页与投稿页可访问
11. PM2 进程存活
12. 健康检查后给出 PASS / WARN / FAIL 总结

#### 推荐实现形式

优先选简单可靠的方式：

- `scripts/ops/domain-smoke-check.sh`
- `scripts/ops/domain-smoke-check.ts`
- `npm run smoke:domain -- --host=https://clawplay.club`

不建议一上来就做复杂监控平台。

先把“上线当天能跑、出了问题能定位”做好就够了。

#### 额外建议

把以下几项也顺手纳入：

- 首页 title / description 是否正确
- 详情页 metadata 是否正确落到对应 Soul
- 分享卡片链接是否没有 404
- 主要静态资源是否能加载

## 7. 风险与取舍

### 风险 1：过早做大 taxonomy

如果现在把分类和标签做得过重，会拖慢后续正常上新。

**取舍：**
只做四类主分类 + 中等规模词表，不做多级树。

### 风险 2：标签完全开放后失控

如果允许公开自由造标签，后续一定会污染索引和 SEO。

**取舍：**
允许提议，不允许直接公开生效。

### 风险 3：一次性清洗所有旧数据太重

如果标准太细，容易变成长期整理工程。

**取舍：**
先做审计表，再做高优先级问题修复，不追求一次把所有文案磨到极致。

### 风险 4：域名检查脚本做得过复杂

如果脚本直接扩张成运维平台，得不偿失。

**取舍：**
只做上线 smoke check，不做长期监控系统。

## 8. 实施拆解

推荐按这个顺序推进：

### P1. 分类与来源解耦

1. 明确最终结构：`category` 只保留用途维度
2. 在文档中把 `translated` 标记为迁移项
3. 设计前台“来源筛选”和“翻译精选”承接方式
4. 再进入代码迁移

### P2. 标签治理升级

1. 扩充正式标签词表到 35–45 个
2. 补齐 alias / synonym 规则
3. 投稿模型新增 `proposedTags`
4. 后台新增标签归并与收录动作

### P3. 存量 Soul 二次清洗

1. 做内容审计脚本或审计报告输出
2. 统一修 metadata 与 preview 三件套
3. 统一修 translated 原文排版与授权说明

### P4. 域名上线工程化

1. 把 `06` 文档收口成可执行步骤
2. 新增 smoke check 脚本
3. 增加 `npm script`
4. 域名放开当天跑一次正式验收

## 9. 验收标准

满足以下条件，可视为本阶段完成：

1. `translated` 是否保留已正式拍板，并写入文档
2. 投稿端与后台都支持“标签提议但不直接公开”
3. 标签词表有正式词、别名词、禁用思路三层约束
4. 现有 8 个静态 Soul 有一份二次审计结果
5. 域名上线可通过一条命令完成 smoke check
6. 出现 FAIL 时能快速定位是 DNS、反代、应用还是页面 metadata 问题

## 10. 上线与回滚

### 上线顺序

1. 先发文档与 schema 调整
2. 再发后台与投稿治理能力
3. 最后发域名 smoke check 工具

### 回滚策略

- `translated` 迁移如出现问题，短期可保留兼容值，不强制删库式回退
- 标签提议机制如影响投稿主链路，可暂时隐藏提议字段，保留后台能力
- 域名 smoke check 脚本不会影响线上业务，可独立回滚
