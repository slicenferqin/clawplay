## 1. 子项边界与关键决策

### 1.1 `22` 是 `P7 / N1` 的独立子项

这份文档只覆盖：

- 产品定位重写
- 全站关键页面文案重构
- 术语体系与 metadata 口径统一

它不覆盖：

- Schema v2
- 人格分析数据结构
- 雷达图组件
- Soul Pack manifest
- 新版人格合集

这些内容仍由：

- `20-soul-as-persona-vnext.md`
- `22-persona-analysis-foundation.md`

分别统筹。

### 1.2 文案改写不等于功能承诺升级

本子项最重要的原则是：

**文案可以提升认知，但不能虚构能力。**

因此：

- 可以说“导入起始 Soul”
- 可以说“替换当前灵魂”
- 可以说“灵魂人格 preset”
- 但不能说“完整 Agent 一键部署”

### 1.3 统一术语优先于局部润色

如果没有统一词表，每个页面都可能改出自己的语言体系。

所以本子项的交付顺序必须是：

1. 先定词表
2. 再定页面 brief
3. 再改页面
4. 最后回归检查

## 2. 术语词表设计

### 2.1 主术语

建议建立 `Copy Lexicon`，至少包含以下词对：

| 场景 | 推荐表达 | 谨慎表达 |
| --- | --- | --- |
| 产品定位 | Soul 人格分享平台 | Prompt 站 |
| 内容对象 | 灵魂 / 人格 / Soul preset | 提示词 |
| 用户动作 | 导入 / 替换 | 一键安装 |
| 文件说明 | 原始 `SOUL.md` | 系统提示词 |
| 长期方向 | Soul Pack / Agent Template（未来） | 完整镜像（当前已可用） |

### 2.2 分层使用规则

#### 定位层

优先使用：

- 灵魂
- 人格
- 人格分享
- Soul preset

#### 解释层

优先使用：

- 边界
- 气质
- 适合谁
- 是否导入

#### 操作层

允许保留：

- 下载原始 `SOUL.md`
- 复制命令
- 备份 / 回滚

## 3. 页面级实施设计

### 3.1 首页 `src/app/page.tsx`

#### 当前问题

- 更像在说“先浏览、再比较、最后安装”
- 目录感强，人格感不足

#### 改写目标

首页要让用户第一眼理解：

- 这是在挑灵魂人格
- 不是在挑一段文本
- ClawPlay 帮你先理解人格，再决定要不要导入

#### 实施项

- 改 metadata `title / description / keywords`
- 改 Journey 3 卡标题与描述
- 调整“按用途浏览”的模块说明，给后续人格型合集留语言空间
- 检查 Hero 与首页模块的术语统一

### 3.2 About 页 `src/app/about/page.tsx`

#### 当前问题

- 项目说明有，但“为什么不是普通 prompt 站”还没讲透

#### 改写目标

明确回答：

- 什么是 ClawPlay
- 什么是 Soul
- 为什么当前先做 Soul 层
- 为什么未来可能走向 Soul Pack / Agent Template

#### 实施项

- 重写顶部标题与 lead
- 新增或重写“边界说明”部分
- 补“现在不是完整镜像平台”的准确解释

### 3.3 投稿页 `src/app/submit/page.tsx`

#### 当前问题

- 更强调“你现在就可以投 `.md`”
- 但还没有强调“你在分享一份人格 preset”

#### 改写目标

让投稿人理解：

- 你提交的是一个灵魂人格起点
- 即使只有原始 `SOUL.md`，也值得先投
- 展示层文案可以后补，核心是把人格资产先沉淀进站里

#### 实施项

- 重写页面标题、副标题
- 重写投稿说明和审核提示语气
- 明确“投稿的是人格 preset，不是炫技文案比赛”

### 3.4 安装页 `src/app/install/page.tsx`

#### 当前问题

- “安装说明”对 MVP 合理，但现在需要更准确表达

#### 改写目标

保留操作准确性，但主叙事改为：

- 导入起始 Soul
- 替换当前灵魂
- 不合适就回滚

#### 实施项

- metadata 文案收口
- 页面主标题与 lead 改写
- “推荐方式”“手动安装”“常见问题”里统一术语

### 3.5 Soul 详情页 `src/app/souls/[slug]/page.tsx`

#### 当前问题

- 操作区仍然偏“下载 / 安装 / 复制”导向
- 还没形成“人格选择”的语言框架

#### 改写目标

在详情页结构大改前，先用文案把方向扳正：

- 先理解这个灵魂是谁
- 再决定是否导入到自己的龙虾
- 原始 `SOUL.md` 是人格证据，不只是下载附件

#### 实施项

- 重写 CTA 文案
- 重写原始 SOUL 说明文案
- 重写侧边安装块的解释性文字

## 4. README 与 Metadata 同步

### 4.1 README

README 必须同步从：

- 中文 `SOUL.md` 目录与分享站

升级到更准确的：

- 中文 `SOUL.md` 目录与人格分享站
- Soul 人格分享平台

### 4.2 Metadata

需要统一检查：

- 首页 metadata
- About metadata
- 投稿页 metadata
- 安装页 metadata
- Soul 详情页 metadata（至少基线表述）

原则：

- 不堆营销词
- 不承诺当前没有的能力
- 明确 Soul / 人格 / 导入之间的关系

## 5. 推荐实施顺序

1. 写术语词表
2. 改 README 与首页 metadata
3. 改首页 / About 页
4. 改投稿页 / 安装页
5. 改 Soul 详情页文案
6. 做一次全站术语回归

## 6. 验证路径

### 6.1 页面级验证

至少确认：

- 首页读起来像人格平台，而不是 prompt 站
- About 明确解释产品边界
- 投稿页让投稿人明白自己在分享什么
- 安装页没有制造“完整镜像一键部署”的误解
- Soul 详情页的文字已经开始转向“人格选择”

### 6.2 语言级验证

检查是否仍存在明显残留：

- “提示词站”
- “系统提示词”
- “一键安装完整 Agent”

如保留，也应只出现在必要技术上下文，而不是主叙事位置。

## 7. 当前不做但要为后续留口

本子项不直接输出这些内容，但文案要为它们留空间：

- 人格维度雷达图
- Soul Pack manifest
- 人格型合集
- 详情页人格结构四段摘要

也就是说，这次文案不能把后续路线讲死。
