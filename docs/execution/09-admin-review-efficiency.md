# 09 Admin Review Efficiency

- Status: Ready
- Depends On: `docs/execution/08-filing-wait-priority-roadmap.md`
- Owner: Codex + slicenfer
- Last Updated: 2026-03-09

## 1. 背景

ClawPlay 的后台审核链路已经能用，但现在更像“信息都在，只能一条条点开看”，而不是一个真正帮助管理员判断优先级的工作台。

随着 Soul 数量继续增长，这个问题会被快速放大：

- 缺来源的翻译 / 改编稿件容易混在普通列表里
- 有阻断项的稿件需要点进详情才能看出来
- 已经满足发布条件的稿件没有被单独拎出来
- 列表信息太平，管理员很难在 10 秒内判断“先处理谁”

这份文档要解决的核心问题是：**把后台列表页从“记录页”升级成“审核优先级工具”。**

## 2. 目标

本阶段目标：

1. 让管理员一进入后台就能看到当前队列最值得先处理的稿件。
2. 让“缺来源 / 有阻断 / 待发布”三类稿件可以一键切出来。
3. 让列表页本身就展示足够多的风险信息，减少无效点进详情的次数。
4. 让详情页也补上统一的优先级语言，保证列表页与详情页判断口径一致。

## 3. 非目标

本阶段不做：

- 复杂多维排序器
- 批量审核动作
- 多管理员协作锁定
- 自动 AI 审核结论
- 独立 CMS / 中台系统

本轮只做第一版高价值、低复杂度的后台效率优化。

## 4. 核心决策

### 决策 1：引入“审核快捷视图”而不是继续堆更多筛选项

后台需要的不是更多表单控件，而是更直接的工作入口。

因此第一版快捷视图固定为：

- 全部队列
- 只看缺来源
- 只看阻断项
- 只看待发布

这 4 个视图比继续扩充普通筛选器更贴近真实审核动作。

### 决策 2：默认按审核优先级排序

管理员大部分时间不是在找“最新一条”，而是在找“最该先处理的一条”。

因此列表默认排序调整为：

1. 缺来源的翻译 / 改编稿件
2. 有阻断项的稿件
3. 已通过且满足发布门槛的稿件
4. 其他常规稿件

这样后台首页天然更像工作台，而不是数据库浏览页。

### 决策 3：列表页直接暴露风险摘要

每条投稿卡片至少直接展示：

- 当前状态
- 审核优先级
- 来源状态
- 阻断 / 警告 / 通过数量
- 下一步建议

这样管理员不用先点详情，列表页就能完成第一轮分诊。

### 决策 4：详情页复用同一套优先级判断逻辑

列表页和详情页如果说的是两套语言，审核会很快失去一致性。

因此：

- 优先级判断逻辑抽到共享模块
- 详情页顶部同步展示“审核优先级”
- 列表与详情共用同一套来源状态、风险判断口径

## 5. 交付内容

本阶段交付包含：

1. 共享审核队列评估模块
2. 快捷视图与优先级排序
3. 审核列表卡片化改版
4. 详情页优先级补充展示
5. 一份实施文档与验证记录

## 6. 完成标志

满足以下条件即可视为本阶段完成：

1. 后台列表可一键切到“缺来源 / 阻断项 / 待发布”。
2. 列表默认顺序已经体现审核优先级。
3. 每条投稿在列表页就能直接看到风险摘要与下一步建议。
4. 详情页顶部能看到统一的审核优先级提示。
5. 本地构建通过，后台页面回归可访问。
